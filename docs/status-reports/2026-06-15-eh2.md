**Resume here:** The 8 API services no longer each re-derive the same plumbing. Three extractions,
all pure refactors with no behavior change:

1. **`apps/api/src/util/dates.ts`** — `todayStr` / `toISO` / `toDateStr`. Replaced 5 hand-copied
   `toDateStr` converters (transfer, envelope-transfer, reconcile, recurring, transaction), 2
   `toISO` copies (account, envelope), and the `todayStr`/`today` copies in account + recurring +
   `server.ts`. A **dead `todayStr`** in the transaction service was removed (defined, never used).
2. **`apps/api/src/util/groupBy.ts`** — a generic, insertion-order-preserving `groupBy(items,
   keyOf, valueOf)`. Replaced the "fetch children → bucket into a `Map` by parent id → attach"
   loop in the transaction service (×2: allocations + transfer-leg counterparts), the template
   service, and the recurring service.
3. **`apps/api/src/constants.ts`** — `DEFAULT_HOUSEHOLD_ID` moved out of `db/migrate.ts`. All 8
   services **and** the migrator now import it from `../constants`, breaking the coupling of the
   household constant to the migration module (the review's finding).

## Design notes / decisions

- **`util/` directory (new).** `dates` and `groupBy` are framework-free pure utilities used by both
  the service layer and `http/server.ts`. Putting them under `apps/api/src/util/` (rather than in
  `services/`) avoids an `http → services` dependency for a generic helper and matches the
  pure-library spirit of ARCHITECTURE §3.
- **`constants.ts` (not folded into `config.ts`).** `config.ts` is the validated *env* config
  (zod); `DEFAULT_HOUSEHOLD_ID` is a fixed compile-time constant, so a separate module is clearer
  and keeps the env schema focused. When multi-household (#19) lands, the household id becomes a
  request-derived value and this constant goes away.
- **`groupBy` signature.** `(items, keyOf, valueOf) => Map<string, V[]>` fit all four call sites
  cleanly (key is always a string id; value is the per-row view object). Insertion order is
  preserved so an upstream `order by position` survives — important for template/recurring lines.
- **Out of scope (left intentionally):** `server.ts` still has its own `DATE_RE` and a
  `parseMoney`-in-try/catch (`parsePositiveMagnitude`) that EH1's `tryParseMoney` could simplify —
  not part of EH2's "service plumbing" remit; candidate for a later tidy.

## Design notes / decisions

- **`formatCents` stays in the web** (not deleted as the review's wording suggested). It adds the
  currency symbol + locale grouping — presentation, not penny-exact domain logic. The domain's
  `formatMoney` is the canonical *plain-decimal* form (used for input-field values); `formatCents`
  is the display wrapper. Keeping it respects the pure-core / thin-presentation boundary.
- **`tryParseMoney` vs `parseMoney`.** The UI needs to parse partial/typing-in-progress input
  without throwing; the boundary must reject loudly. Same regex, two contracts: `parseMoney` now
  delegates to `tryParseMoney` and throws on `null`. One `DECIMAL_RE`.
- **`formatMoney(value: number)` widening.** It had no production callers (the API returns raw
  integer cents; the web formats), so widening to accept an integer-cents `number` is safe and
  makes the package web-consumable without retyping every `amountCents: number` view to `Cents`
  (that larger retype is out of EH1's scope).

`npm run format`. `@budgeteer/domain` is a workspace package consumed via the hoisted symlink in
root `node_modules/@budgeteer/`; after adding it to `apps/web/package.json` run `npm install` so the
link/lockfile update (Vite then re-optimizes deps on first `dev`). The web imports domain by its
package name (`moduleResolution: Bundler` resolves `exports` → `./src/index.ts`, the raw TS).

**Resume here:** A split can now mix **spend** and **refund** rows in one transaction. A per-row
**Refund** toggle in the allocation editor flips that row **opposite** the transaction's
direction — e.g. a `−$70` receipt = `−$100` Groceries + a `+$30` Refund to Gas (net `−$70`).
The key finding: the split invariant (`validateAllocations`) was **always** defined on the
**signed total**, so it already admitted mixed-sign rows — only the API/UI forced every row to
the txn's sign. So `#8` was a **boundary + UX change, no schema change, no domain change**: an
allocation input gains an optional `refund` flag; the editor nets refund rows and blocks a
net-direction-flip / over-allocation (client + server). Resolves the PRD §9 "negative rows"
open question (FEAT-008). **Next: recurring `#9`**, reconcile `#10`, then analysis (`#11`–`#14`).

## Commands & gotchas (cold-start)

Same as #01–#07. New: allocation inputs accept an optional **`refund: boolean`** (default
`false`) on `POST /accounts/:id/transactions` and `PUT /transactions/:id/allocations`; a refund
row is stored **opposite** the transaction direction. The split invariant is on the **signed
total** (`0 ≤ |Σ| ≤ |amount|`, in the txn's direction) — mixed-sign rows are fine, a net
direction-flip is a `400`. **No schema change** (allocations were already signed).

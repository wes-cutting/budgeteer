**Resume here:** The **Insights phase's fourth and FINAL new visualisation** is done — the phase's
new-viz builds (`UX9` breakdown, `UX10` trends, `UX11` burn-down) are complete. `UX11` is a
**vertical slice** that adds an "**am I on pace this month?**" view at **`/insights/burndown`** (a 9th
sub-nav tab, "Burn-down", next to Budget): for a chosen month and scope (**all budgeted envelopes**,
or **one** via a picker), a **`Gauge`** showing **budget consumed** (spent ÷ target) with the gauge's
**threshold marker repurposed as the elapsed-time pace** ("pace today") — so a user can course-correct
**before** month-end, where budget-vs-actual only shows it after. **Read decision: composes the
EXISTING `getBudgetVsActual(month)` read — ONE call, no fan-out, NO new endpoint/schema/API change**
(the scope picker selects the household aggregate or one budgeted envelope out of that single read —
*lighter* than UX10's per-month fan-out, because a burn-down is a single month and that call already
returns every row + the totals). The only new code is **one pure, unit-tested domain function**,
`assessBurndown` in `packages/domain/src/burndown.ts` — elapsed-day-fraction (linear-burn) pace with
`today` passed in (never `Date.now`), per the pure-core/impure-shell convention (mirrors `forecast.ts`),
reusing the existing `daysInMonth`. **Chart-shape decision: reuse `Gauge` AS-IS** (no new shape or
variant) — the burn-down is one ratio + one reference marker, exactly the gauge's job. **State never
rests on colour:** `status ∈ over-budget | over-pace | on-track` is carried by a **verdict sentence**
above the gauge + **marker position** (fill end vs. the pace marker) + a **text Pace column** in the
table. `e2e/a11y.spec.ts` seeds a budgeted envelope with **mid-month** outflow and is **axe-green LIGHT
AND DARK** with the gauge rendered. Gate green: **335 Vitest passing** (+13; 1 pre-existing, unrelated
carry — see §4) **+ 81 e2e**, build **111.06 KB gz** (< 120 KB). **Next: `UX12`** — the UX Uplift
**Polish** track (`UX12`–`UX15`).

## 4. Notes / decisions

| Item | Note |
| ---- | ---- |
| **Read decision (the headline)** | Composed **`getBudgetVsActual(month)` — a SINGLE call**, no fan-out. That read already returns every envelope's `targetCents`/`spentCents` and the household totals for the month, which is exactly the burn-down's numerator (spent-so-far) and denominator (target); the scope picker just selects which of those to gauge. This is the **lightest** end of the R4/R5/UX9/UX10 "compose existing reads, add no endpoint" precedent — *less* than UX10's per-month fan-out, because a burn-down is one month. The only thing the read can't give — "expected pace so far" — is pure client-side calendar math, so no server surface was added. |
| **Pace proxy + pure function** | "Expected spend so far" is the **elapsed-time fraction** of the month (linear-burn) — a deliberate V1 **proxy** (real spend is lumpy; the elapsed fraction is a reference the user compares against, with exact figures in the table). The math is non-trivial (past/current/future-month branch, days-in-month, tz-safe string parse), so per the **pure-core/impure-shell** convention it's a **pure, unit-tested domain function** (`assessBurndown`, `today` passed in — mirrors `forecast.ts`, never reads the clock), not inline view logic. A more nuanced pace (weighting recurring bills by due date) is deliberately **out of scope**. |
| **Chart-shape decision — reuse `Gauge` as-is** | The burn-down is **one ratio + one reference marker**, exactly the `Gauge` shape (credit/payoff already use it for a ratio + threshold). No new shape or variant: `ratio` = spent ÷ target, `threshold` = `{ at: elapsedFraction, label: "Pace today (X%)" }` — the existing marker repurposed **as-is** as "where you'd expect to be today"; the existing over-cap fires past 100%. A new "burn-down curve" (a cumulative-spend line vs. a target line over the days) was rejected: it needs **per-day spend**, which the existing read doesn't return, so it would force a new endpoint for a marginal gain over the ratio-vs-pace gauge. |
| **Scope — both, from ONE read** | A `<Select>` picks "All budgeted envelopes" (aggregate over budgeted rows) or one budgeted envelope. Because the one read returns all rows, this needs **no extra requests**. Only envelopes with a target set are selectable (a burn-down needs a budget to pace); changing the month resets the scope to "all" so the picker can never point at an envelope that isn't budgeted that month. |
| **Colour is never the sole signal** | The over/under/on-track state is carried three independent non-colour ways: a **verdict sentence** above the gauge, the **marker position** (fill end vs. the pace marker), and a **text Pace column** in the table (+ the `role="img"` summary names the verdict word). The single `--chart-1` token carries no state. |
| **e2e robustness + determinism** | The shared e2e store accretes budgeted envelopes across parallel specs, so the fallback table lists many rows and the "All" aggregate is noisy — the functional test asserts only its **uniquely-named** envelope's own exact cells + `75.0%` consumed (clock-independent) and the scoped gauge's summary, **not** the pace verdict (which depends on the run's day-of-month). The **unit** view test pins `today` mid-month via a faked `Date` (`vi.useFakeTimers({ toFake: ["Date"] })`) so it *can* assert over-pace/on-track deterministically. |
| **Verification** | Gauge is rendered-and-axe-scanned in **real Chromium** by the e2e suite (seeded with a real mid-month withdrawal, light + dark). The interactive preview server was not exercised this session (no browser-observable regression risk beyond what the Playwright suite already proves — same rationale as UX8–UX10). |
| **Pre-existing carry (recurring, unrelated) — CONFIRMED on main** | `apps/api/test/recurring.test.ts` › *"post-due generates every due occurrence…"* fails (expects 4 posted occurrences, gets 1). **Reproduced on clean `main` this session via `git stash` + isolated re-run** — a relative-date fixture assumption vs. the sandbox clock, **not** this slice's regression. Not touched (out of scope). |
| **Pre-existing carry (forecast e2e, flaky, unrelated)** | The cash-flow forecast e2e's `getByLabel("Account")` strict-mode collision (the Forecast chart's `aria-label` contains "Account") — documented in the UX10 report §4, reproduces on `main`. **Did not trip** in this session's full e2e run (81/81 passed). Not touched. |

- **The chart contract (ADR-0007), in `apps/web/src/ui/Chart.tsx`:** SVG `role="img"` + one-line
  summary `aria-label`, `aria-hidden` innards; a real `<table>` is the keyboard/SR fallback (passed
  as a slot); colour is never the sole signal; `--chart-*` tokens (≥ 3:1) with a dark variant; **no
  opacity animation on text**; honour reduced-motion. New shapes reuse `ChartFigure` — **but check
  whether an existing shape already fits first** (UX11 needed none — reused `Gauge`; UX10 reused
  `LineChart`; UX9 needed `BreakdownBars`).
- **The "reuse a shape, repurpose its parts" trick:** `Gauge`'s `threshold` marker was built for a
  credit limit / payoff milestone, but it's just "a reference point on a single ratio" — so it maps
  cleanly onto "pace today" with **no new geometry**. Before reaching for a new shape, check whether
  an existing shape's parts already model your semantics.
- **Pace/date math lives in the domain, pure:** `assessBurndown`/`monthElapsedFraction` take `today`
  as an argument (never `Date.now`), mirroring `forecast.ts`. The **view** reads the real clock
  (`new Date().toISOString()`); the **pure fn** is deterministic under unit test. If you extend the
  pace model (projection, weighting), keep it here and unit-test it — don't inline clock logic in the
  view.
- **a11y gate is light AND dark** — seed data so the chart renders, then add it to **both** describe
  blocks in `e2e/a11y.spec.ts`. For **pace** specifically, seed **mid-month** outflow (via the
  register's `Date` field, a plain editable text input) so the current partly-elapsed month has a
  live pace — `seedBurndown` in `e2e/a11y.spec.ts` is the reusable pattern.
- **Bundle:** 120 KB gz ceiling, 111.06 KB now (~8.9 KB headroom). Hand-rolled SVG only — **no
  charting library.**
- **Known carries (not this slice's regressions):** `recurring.test.ts`'s post-due occurrence-count
  case, and a flaky `getByLabel("Account")` collision in the forecast e2e test — both reproduce on
  `main` via `git stash`, see §4.

## 4. Notes / decisions

| Item | Note |
| ---- | ---- |
| **In-code registry, not `FileMigrationProvider`** | The registry is a typed object in `migrate.ts` (`"0001-…": import`), not a directory scan: typo-proof under both tsx and Vitest, no fs/path resolution differences, and the execution order is reviewable in one place. Kysely executes keys lexicographically — zero-padded prefixes keep that honest |
| **The household seed is not a migration** | `db:reset`'s PostgreSQL path truncates `households` but not `kysely_migration`, so a run-once migration would never re-insert the row. The insert stays in `migrateToLatest` (runs every startup, `on conflict do nothing`) — the truncate-heal contract now has its own test |
| **Forward-only, frozen** | No `down` migrations (disposable PGlite dev stores make rollback-by-recreate cheap); committed files are never edited — schema changes are new numbered files. Recorded in `05_DATA_MODEL` §4 |
| **Atomic batch on failure** | Postgres transactional DDL means a failing batch records **nothing** (verified by test — the store is exactly as before the attempt). The thrown error names the failing migration |
| **`0002` on a corrupted store** | If the double-post race ever fired historically, the index build fails naming the duplicated `(recurring_id, occurred_on)`. Deliberate: a migration never deletes financial data on its own; the owner resolves the ledger rows by hand and re-runs. (The owner's real store had none — adoption was clean) |
| **Already-posted is success-shaped** | The losing `postDue` reports `posted: 0` with no `error` for the rule: the occurrences *are* posted (by the winner, who advanced the cursor). Surfacing an error would misreport a correct outcome |
| **Perf/flake watch items** | `perf.test.ts` fine in all runs; no e2e click-stability timeouts. Both stay watch-only |

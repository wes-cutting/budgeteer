## What R14 did

**No spike — no risk.** R14 is pure test infrastructure: the coverage gap (pre-#11 features
not covered by a real browser test) was identified by the 2026-06-17 improvement review, and the
approach (per-area split + shared helpers) was straightforward. The only unknowns were UI locator
details (discovered by reading the source components).

**Three-part delivery:**

1. **`e2e/setup.ts`** — shared helpers reused by multiple spec files: `createAccount` (fills and
   submits the Add account form, asserts the account button appears), `createEnvelope` (same for
   envelopes), `goToDashboard` (clicks ← Dashboard if visible). Each spec imports only what it
   needs.

2. **8 per-area spec files** (all new; `journey.spec.ts` deleted):
   - `e2e/accounts.spec.ts` (2 tests): dashboard CORS check + create account + open register.
   - `e2e/envelopes.spec.ts` (1 test): create → archive → unarchive full lifecycle (#6).
   - `e2e/transactions.spec.ts` (5 tests): single-allocation deposit (core loop, #3) · split
     allocation (#3) · partial/needs-allocation (#3) · edit-split (#5) · refund row (#8).
   - `e2e/templates.spec.ts` (2 tests): save-as-template + apply-to-second-transaction (#4) ·
     create-from-TemplatesView + rename (#4).
   - `e2e/transfers.spec.ts` (2 tests): account transfer (#7a) · envelope reallocation (#7b).
   - `e2e/recurring.spec.ts` (1 test): create monthly rule + post due (#9).
   - `e2e/reconcile.spec.ts` (2 tests): reconcile with matching balance · reconcile with
     mismatched balance shows the difference (#10).
   - `e2e/analysis.spec.ts` (5 tests): spend by envelope grid (#11) · budget vs. actual with
     inline target PUT (#12) · cash-flow forecast with expected-spend toggle (#13) · credit
     utilization with limit PUT (#14a) · debt payoff with principal PUT (#14b).

3. **`journey.spec.ts` deleted** — its 9 chained steps are now distributed across the area specs
   above (steps 1-5 → `accounts`/`transactions`; steps 6-9 → `analysis`). The previous single
   test counted as "1 e2e" — the new suite counts as **20 e2e**.

**Three locator lessons during the run:**

- The recurring-rule list shows `accountName · direction · amountCents · frequency · lines
  (envelopeName)` — not the payee. Assert on the ENVELOPE name, not PAYEE.
- `getByRole("button", { name: "Save" })` matches both "Save template" and the inline "Save" rename
  button; `{ exact: true }` disambiguates.
- After a reconciliation, both the `role="status"` notice and the "Last reconciled" paragraph
  contain "off by $10.00"; use `.first()` or rely on the status assertion alone.

## Commands & gotchas (cold-start)

Gate is unchanged: `npm run typecheck`, `npm run lint`, `npm test`, `npm run test:e2e`,
`npm run format`, `npm run build --workspace @budgeteer/web`.

- `npm run test:e2e` needs Chromium once (`npx playwright install chromium`); it boots both
  servers (web on `:5173` to match the API CORS allowlist) and now runs **20 tests across 8
  spec files** in 1 worker (serial, ~9 seconds).
- Each spec uses `Date.now()` **inside each `test()` function** for stamps — so tests within the
  same spec file are isolated even though they share the same stamp variable if defined at
  file scope. Each test creates its own uniquely-named entities.
- The recurring test anchors the rule at `2020-01-01` (a guaranteed past date) so it is
  immediately due when "Post due" is clicked.
- `reuseExistingServer: !process.env.CI`: locally, the in-process PGlite accumulates data across
  runs. Stamps ensure isolation; tests assert only on entities they created.
- **Convention going forward:** every new slice (R6–R9, #15+) lands with its own `e2e/*.spec.ts`
  covering its feature journey. Add shared setup helpers to `e2e/setup.ts` as needed.
- CI (`.github/workflows/gate.yml`) stays **manual-only** (`workflow_dispatch`) — e2e CI
  automation is `#16`, not now.

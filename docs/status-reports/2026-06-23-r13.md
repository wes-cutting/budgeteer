`buildServer`'s `logger` option is widened from `boolean` to `FastifyServerOptions["logger"]`; `index.ts`

type/message/stack) — **never** bodies or headers — so R13 keeps them and adds **no** body serializer; the
regression net asserts both that the access line is present *and* that a money-route body + `authorization`
header never leak. **This completes DX `R10`–`R13`. The roadmap's DX + UX backlog is now clear — remaining
work is the Hardening track and new feature slices.**

## Challenge findings (before building)

The kickoff named the riskiest bit correctly: logging is a **cross-cutting effect, not a route**, and the
real risk is leaking secrets/request-bodies (SECURITY.md §1/§5) while still getting useful lines. I read
`config.ts`, `http/server.ts`, `index.ts`, the `backup`/`transactions` routes, and Fastify's own logger
types/source, and confirmed:

- **Riskiest assumption — what gets logged.** Verified against `node_modules/fastify/lib/logger-pino.js`
  and `types/logger.d.ts`: Fastify's **default** serializers emit `req → {method, url, host, remoteAddress,
  remotePort}`, `res → {statusCode}` (+ `responseTime`), `err → {type, message, stack}`. **No request body,
  no headers.** So financial bodies — a transaction's `memo`/`payee`/`amount`, and the `/export` snapshot
  (which rides in the *response body*) — never reach the logs **provided R13 adds no custom serializer**.
  This API also puts no secrets in URLs/query strings, and the error handler already logs only the error
  object (`app.log.error(e)`, 5xx). The mandate was therefore narrow: **enable the built-in logger, change
  no serializer.**
- **Decision — where `LOG_LEVEL` is read + validated.** It already lived in `config.ts` as
  `z.string().default("info")` but was **unused**. Tightened it to `z.enum([...pino levels]).default("info")`
  so an invalid level fails loudly at the **config boundary** (fail-loud, spine §8) rather than throwing on
  the first log. `index.ts` reads `config.LOG_LEVEL` and threads it into `buildServer`.
- **Decision — minimal `buildServer` enrichment.** Widened `logger?: boolean` →
  `logger?: FastifyServerOptions["logger"]` (the Fastify logger option itself). Production passes
  `{ logger: { level } }`; the default stays `false`, so `createTestApp()` and every existing test are
  **byte-for-byte unchanged** (quiet + deterministic). The wider type also lets the leak test inject a
  capture stream — no new bespoke parameter on `buildServer`.
- **Regression net (asserts both halves).** A new `logging.test.ts` builds the server with
  `logger: { level: "info", stream }`, POSTs a real money-route body carrying unique markers
  (`memo`, `payee`, a distinctive amount) **and** an `authorization` header, then asserts: (1) the access
  line **is** emitted (`incoming request` / `request completed`, the url, `"statusCode":201`) — logging
  stays on; and (2) the captured output does **not** contain any marker or the word `authorization` — no
  leak. A `config.test.ts` adds the fail-loud boundary proof (default → `info`; valid level accepted;
  unknown level throws). A real-startup smoke (`LOG_LEVEL=debug npm run start`) confirmed structured JSON to
  **stdout** with a `reqId` correlation id and a clean leak grep.

**Decision:** widen the `logger` option, thread the validated `LOG_LEVEL`, keep Fastify's default
serializers (add no body serializer), and pin it with a capture-stream leak test + a config-boundary
fail-loud test. No API-contract/schema/domain change.

## What R13 built

### Changed: `apps/api/src/config.ts`

`LOG_LEVEL` tightened from `z.string().default("info")` to
`z.enum(["fatal", "error", "warn", "info", "debug", "trace", "silent"]).default("info")` — a closed set,
validated at the boundary, fail-loud on a typo.

### Changed: `apps/api/src/http/server.ts`

`buildServer`'s `opts.logger` type widened `boolean` → `FastifyServerOptions["logger"]` (import added). A
comment documents the guardrail: keep pino's default serializers, add **no** body serializer, so bodies and
headers never reach the logs (SECURITY.md). `Fastify({ logger: opts.logger ?? false })` is otherwise
unchanged — `false` remains the default for tests.

### Changed: `apps/api/src/index.ts`

`buildServer(db, { logger: true, ... })` → `buildServer(db, { logger: { level: config.LOG_LEVEL }, ... })`.
The only behavioral change: production logging now respects `LOG_LEVEL` (was hard-coded to pino's `info`).

### New: `apps/api/test/logging.test.ts` (+2) · `apps/api/test/config.test.ts` (+3)

The capture-stream leak/access test and the config-boundary fail-loud test (above).

**No API-contract / schema / domain change.** No `06_API_CONTRACT.md` / `05_DATA_MODEL.md` /
`04_DOMAIN_MODEL.md` edit. No `apps/web` change. No new dependency (pino ships inside Fastify).

## Commands & gotchas (cold-start)

Gate: `npm run typecheck && npm run lint && npm test && npm run test:e2e && npm run format && npm run build --workspace @budgeteer/web && npm audit --omit=dev --audit-level=critical`

- **Logging is OFF in tests.** `buildServer(db)` defaults `logger: false`; only `index.ts` (and the two new
  tests) enable it. Keep it that way — turning it on in `createTestApp()` would make the suite noisy and
  non-deterministic.
- **Never add a body/header serializer to the Fastify logger.** Fastify's defaults log method/url/status
  only; a custom `req` serializer that includes `req.body`/headers would leak financial data (SECURITY.md
  §1/§5). The leak test in `logging.test.ts` is the guard — if you change the logger config, that test must
  stay green.
- **`LOG_LEVEL` is a closed enum.** Valid values: `fatal|error|warn|info|debug|trace|silent` (default
  `info`). Anything else fails loudly at startup. Set it in `.env` / the environment for dev-vs-prod
  verbosity.
- **SCA note:** `npm audit` reports 3 **high**-severity advisories (kysely chain); the gate is
  `--audit-level=critical`, so it passes (exit 0). Pre-existing, not introduced by R13.
- `reuseExistingServer: !process.env.CI` — kill stale `:3001`/`:5173` before e2e if you ran the API or preview.

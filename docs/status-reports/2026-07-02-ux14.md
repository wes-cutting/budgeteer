allocated", "No forecast yet"…) — a fragmented, un-guided first impression. The home now **DERIVES**

metric. The instant **anything** exists the cockpit takes over (its per-panel empty states guide the
rest — progressive disclosure). A11y: a **named region** (`aria-label="Get started"`) under the home's
`<h1>`, an **ordered `<ol>`** two-step list, accent CTA links clearing ≥ 4.5:1 light+dark; order/state

## 4. Notes / decisions

| Item | Note |
| ---- | ---- |
| **ADR-or-not / spike-or-not** | **No ADR, no spike, no new dependency.** `FirstRunOnboarding` composes the UX4 `EmptyState` + React Router — additive presentation on a validated foundation, no data/API/domain change. §11 fast-path; the FEAT note is the paperwork. |
| **Derive-don't-store** | First-run is `accounts.length === 0 && envelopes.length === 0` derived from existing reads — **no new server "has onboarded" flag**. No second source of truth to drift; self-correcting when data appears. This was the kickoff's named risk and is honoured. |
| **Replace vs. prepend** | On a truly-empty app the guided screen **replaces** the cockpit grid (a first-run app should feel guided, not be a dashboard of five empty panels). Once any data exists the full cockpit returns — no half-onboarded state. |
| **Test-safety on the run-scoped store** | The e2e store **accumulates across the whole run** (per-run throwaway PGlite, no per-test reset; tests use unique names). `a11y.spec.ts` sorts first and its empty-store scans are placed **first in the file**, so they hit a genuinely empty app. All other journeys `createAccount`/`createEnvelope` first (landing on `/accounts`·`/envelopes`), so they never trip onboarding on the home. |
| **Same-change test consequence** | `Home.test.tsx`'s empty-app case previously expected the cockpit **Overview** region on a default (empty) fake API; it now expects the onboarding region, with a seeded case added for the cockpit branch. `Cockpit.test.tsx` is untouched (it still renders `<Cockpit>` directly, empty, with all five panels). |
| **Pre-existing carry (recurring, unrelated) — RE-CONFIRMED on clean `main`** | `apps/api/test/recurring.test.ts` › *"post-due generates every due occurrence…"* fails (expects 4 posted, gets 1). Reproduced on clean `main` this session (tree clean at start; isolated re-run 4 passed \| 1 failed). A relative-date fixture vs. the sandbox clock — **not** this slice's regression (this slice is web-only). Not touched. |
| **Pre-existing carry (forecast e2e, flaky, unrelated)** | The cash-flow forecast e2e's `getByLabel("Account")` strict-mode collision (documented UX10/UX11 §4). **Did not trip** this run (92/92). Not touched. |
| **Verification** | The onboarding surface is rendered-and-axe-scanned in real Chromium (`a11y.spec.ts`, light + dark) with the surface **visible**; the component contract + the derived Home branch are unit-covered. Interactive preview not exercised (same rationale as UX8–UX13 — no browser-observable regression risk beyond the Playwright + axe suites, which render the real surface in both schemes). |

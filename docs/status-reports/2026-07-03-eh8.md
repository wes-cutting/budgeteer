**Resume here:** `EH8` is **done, gate-green** (376/376 Vitest · 94/94 e2e · build 117.32 KB gz).
The policy is decided and recorded in [`04_DOMAIN_MODEL` §6](../04_DOMAIN_MODEL.md): **calendar
dates are user-local** — the client derives "today"/"this month" from the browser's local clock
(new [`apps/web/src/dates.ts`](../../apps/web/src/dates.ts); the 8 UTC-deriving components and the
fakeApi switched off `toISOString()`), and **the server never derives a user-facing calendar
date** — every date/month parameter is now **required** at the HTTP boundary (`400` when
missing/malformed; `06_API_CONTRACT` §1 + every touched endpoint updated). This is a **breaking
API change** (documented; the bundled web app is the only client — its `httpApi` adapter fills
every date param from the local clock in one place). The evening-west-of-UTC bug class is closed:
quick-add defaults, the register's month window, "this month" reads, burndown pace, recurring
due-ness, and the forecast's day zero all follow the user's calendar now. **Next:** the remaining
review items `EH9`/`EH10`/`EH14`, `EH12`/`EH13`, or the sheet-parity features `S7`/`S9` — order is
the owner's call (the review's suggested order continues with `EH10`, the restore proof — check
EH9's "migrator must land before `#15b`" constraint when scoping).

## 4. Notes / decisions

| Item | Note |
| ---- | ---- |
| **Required, not fallback** | The route params could have kept a UTC fallback, but that leaves the bug latent for any future client. EH7's precedent (`todayStr` *requires* the clock) applies one layer up: make the regression impossible. The web `Api` port keeps optional dates for ergonomics; the `httpApi` adapter is the single place local derivation feeds the wire |
| **Breaking change, priced in** | 8 endpoints changed shape (one new required body field on `POST /accounts`). Single bundled client, no versioning promise pre-`/v1` (`06_API_CONTRACT` §1) — documented rather than shimmed. Anyone curl-scripting against the API must now pass dates |
| **`POST /recurring` takes `today`** | Slightly odd on a write, but the response view embeds `dueCount`, which is today-relative. The alternative (drop `dueCount` from the create response) would change the view shape for every reader |
| **Instants unaffected** | `createdAt`/`archivedAt` stay UTC timestamps — they record *when*, not *which budget day*. Deliberately out of scope (same call as EH7's) |
| **e2e false alarm (K20)** | The first gate run failed 6 empty-app a11y specs — the Playwright config's `reuseExistingServer` had reused the owner's running dev stack (real ledger data), so "first-run onboarding" found a populated app. Clean-stack re-run: 94/94. Dev servers were restarted afterwards. Kit lesson filed as K20; **gotcha recorded in §6** |
| **Perf/flake watch items** | `perf.test.ts` didn't trip in any run; no click-stability timeouts this time. Both stay watch-only |

## 6. Commands & gotchas (cold-start)

```sh
npm run typecheck && npm run lint && npm run format
npm test                 # 376 passing
npm run test:e2e         # 94 Playwright — STOP any dev servers on :3001/:5173 first (see below)
npm run build --workspace @budgeteer/web   # 117.32 KB gz
```

- **e2e needs to own the stack:** if a dev API/web server already holds `:3001`/`:5173`, Playwright
  reuses it (`reuseExistingServer`) — against the dev store, whose data fails the 6 empty-app
  onboarding/a11y specs. Stop the dev servers before `npm run test:e2e` (K20 proposes making the
  config fail fast instead).
- **The API now requires every user-facing date** — curl calls need `occurredOn`/`month`/`today`/
  `from`+`to` explicitly (`06_API_CONTRACT` §1). An already-open web tab from before this change
  runs the old bundle and will 400 until refreshed.
- Date-sensitive API tests use **absolute fixture dates** passed explicitly (the EH7 pin
  `createTestApp({ today })` now matters only for the backup-filename test).
- Client code must derive calendar dates via `apps/web/src/dates.ts` — never `toISOString()`.

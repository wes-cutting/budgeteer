**Resume here:** **`#11` is `Done` and gate-green** — the first analysis slice. Budgeteer now has a
**generated** "spend by envelope over time" grid (the replacement for the spreadsheet's hand-keyed
"18 Monthly" tab): rows are envelopes, columns are periods, each cell is the **net signed allocation
flow** (`+` funded / `−` spent). Built data → API → UI with **no schema change** (a read-only
aggregate query). **Next: `#12` — budget vs. actual**, which is **gated by a capability that does not
exist yet — per-envelope monthly targets** (PRD §9 open Q). The kickoff prompt at the bottom frames
that as the first thing to settle with the owner. Gate, money rules, and CORS notes are unchanged.

## What #11 did

**The modeling question (settled with the owner before building).** "Spend by envelope over time" is
a *definition*, not just code, so it was put to the owner first (challenge-the-plan):

- **Net signed allocation flow** per cell — one signed number (`+` funded, `−` spent), not split
  spent/funded columns. (Mirrors the "18 Monthly" grid and how envelope balances derive.)
- **Exclude** envelope↔envelope reallocations — count only real transaction flow (allocations).
  Reallocations are internal budget maintenance, not income/spend.
- **Monthly grid + annual rollup** (a grain toggle).
- Decided in passing (definitions, not forks): bucket by the **calendar period of the transaction's
  `occurred_on`** (a `DATE`, no timezone); **archived envelopes included** (history preserved);
  only envelopes/periods **with activity** appear (gap-fill is a noted fast-follow). These are
  captured in [FEAT-011](../features/analysis-envelope-spend.md) §11 — **no ADR** (it's a definition,
  not a reversible architectural decision).

**No spike.** The slice derives entirely from existing, already-validated data
(`allocations`/`transactions`); no external source, integration, or perf unknown at V1 volumes — so
it went straight to spec → slice (stated explicitly when challenging the plan).

**The build (vertical):**

- **Data/API** — new [`analysisService.ts`](../../apps/api/src/services/analysisService.ts):
  one household-scoped Kysely aggregate over `allocations ⋈ transactions ⋈ envelopes`, period via
  `to_char(t.occurred_on, 'YYYY-MM' | 'YYYY')`, assembled into a period-aligned grid with row/column/
  grand totals using the EH2 `groupBy`. Integer cents summed in SQL, narrowed at the boundary. New
  route `GET /analysis/envelope-spend?grain=month|year` (zod-style closed-enum check at the boundary →
  `400`; default `month`). **No table, no view, no migration.**
- **UI** — new thin [`AnalysisView.tsx`](../../apps/web/src/AnalysisView.tsx): a real `<table>`
  (`<caption>`, `scope="col"` period headers, `scope="row"` envelope headers, totals `<tfoot>`), a
  grain toggle (radiogroup), and loading/empty/error/success states; reached from a Dashboard
  **Analysis** button. Money shown via the web `formatCents` (EH1). Never touches the datastore.

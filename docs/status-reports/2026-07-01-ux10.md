**Resume here:** The **Insights phase's third new visualisation** is done. `UX10` is a **vertical
slice** that adds a "**is my spending going up or down?**" view at **`/insights/trends`** (an 8th
sub-nav tab, "Trends", next to Breakdown): the household's **total outflow** plus its **top-2
envelopes by outflow**, as a multi-series **line chart** over a selectable **3/6/12-month trailing
window**. It **reuses the existing `LineChart` shape** on the `ui/Chart` primitive — no new chart
shape was needed (unlike UX9's `BreakdownBars`). **Read decision: composes the EXISTING
`getBudgetVsActual(month)` read, once per month in the window (`Promise.all`) — NO new endpoint /
schema / API / domain change**, extending the R4/R5/UX9 fan-out precedent to *several* calls because
that read is per-month only. It charts **pure outflow** (`spentCents`), not `getEnvelopeSpend`'s
*net* flow, for the exact reason UX9 already established: an envelope funded and spent in the same
month would understate true outflow, and calling a net-flow line a "spending trend" would be
actively misleading. **Legibility without relying on colour:** three series max, matching the
**3 `--chart-*` tokens 1:1** (no colour reuse) — Total (solid/circle) + the two ranked envelopes
(dashed/square, dashed/triangle) — each also carrying a direct end label; envelopes with zero
outflow across the whole window are excluded from ranking, and a ranked envelope's zero-spend month
is **0-filled** (not omitted) so the axis stays aligned. `e2e/a11y.spec.ts` now **backdates real
withdrawals into two different months** and is **axe-green LIGHT AND DARK** with the chart rendered.
Gate green: **323 Vitest** (1 pre-existing, unrelated carry — see §4) **+ 78 e2e**, build
**109.90 KB gz** (< 120 KB). **Next: `UX11`** — new viz: budget burn-down.

## 4. Notes / decisions

| Item | Note |
| ---- | ---- |
| **Read decision (the headline)** | Composed **`getBudgetVsActual`**, called once per month in the window via `Promise.all` — **not** `getEnvelopeSpend` (a single call, but *net* flow). Both are existing reads (no new endpoint either way), but net flow would mix funding into a "spending" line — the same problem UX9's kickoff already reasoned through for the breakdown numerator, now extended to a trend. The cost is N parallel requests instead of one; negligible at V1's single-household, bounded-window scale. |
| **Chart design — Total + top-2, one combined chart** | Considered per-envelope small multiples (a mini chart per envelope) — rejected: it would need a new chart shape and answers "what's driving it" without answering "is total spending trending up?" as directly. Total + top-2 in **one** `LineChart` answers both, reuses the existing shape untouched, and keeps exactly 3 series = 3 tokens (no colour reuse, unlike a naive "all envelopes" approach which would have forced either colour reuse or an unreadable tangle of lines). |
| **Ranking & 0-fill** | Envelopes are ranked by **total outflow summed over the whole window**, so the top-2 can shift when the window changes (3 vs. 6 vs. 12 months) — intentional; "top spenders" is scoped to what's on screen. An envelope with **zero** outflow across the whole window never appears (no meaningless flat line); a *ranked* envelope's individual zero-spend month is **0-filled**, not omitted, keeping the month axis continuous and aligned across series — mirrors `getEnvelopeSpend`'s own 0-fill convention. |
| **End-month picker (not just "now")** | Added an explicit **End month** control (mirroring Breakdown/Budget's month picker) rather than always trending "to today" silently. This makes the view strictly more useful (review a past window) and — as a side effect — makes it deterministically testable without depending on the test run's wall-clock date, the same testability reasoning that shaped the other Insights views' controls. |
| **e2e robustness** | The shared e2e store accretes outflow across parallel specs, so the household-wide **Total** column isn't deterministic (same issue UX9's report documents). The functional test spends a **deliberately large, dominant** amount on one uniquely-named envelope across two real (backdated) months, then asserts only that envelope's own exact table cells + ascending month order — never the Total column. |
| **Backdating via the register's `Date` field** | `AddTransactionForm`'s `Date` input is a plain editable text field (no min/max), so both the e2e a11y seed and the functional test set it directly to a prior month — the first genuinely **multi-month** e2e data in the Insights a11y suite (prior scans all seeded a single month). |
| **Verification** | Chart is rendered-and-axe-scanned in **real Chromium** by the e2e suite (seeded with real backdated withdrawals, light + dark). The interactive preview server was not exercised this session (no browser-observable regression risk beyond what the Playwright suite already proves — same rationale as UX8/UX9). |

## 6. Commands & gotchas (cold-start)

```sh

                          # unrelated carry — see §4; perf.test.ts may trip under load)
```

- **The chart contract (ADR-0007), in `apps/web/src/ui/Chart.tsx`:** SVG `role="img"` + one-line
  summary `aria-label`, `aria-hidden` innards; a real `<table>` is the keyboard/SR fallback (passed
  as a slot); colour is never the sole signal; `--chart-*` tokens (≥ 3:1) with a dark variant; **no
  opacity animation on text**; honour reduced-motion. New shapes reuse `ChartFigure` — **but check
  whether an existing shape already fits first** (UX10 needed none; UX9 needed `BreakdownBars`).
- **The "3 tokens, 3 series max" trick:** when a trend/comparison could have many series (many
  envelopes), pick a **small, ranked subset** (here: Total + top-2) so every series gets its own
  `--chart-*` token with **no reuse** — reuse this instead of stretching to more series or reusing a
  token across two simultaneously-visible lines.
- **a11y gate is light AND dark** — seed data so the chart renders, then add it to **both** describe
  blocks in `e2e/a11y.spec.ts`. For a **multi-month** chart specifically, seed via the register's
  `Date` field (a plain editable text input, no min/max) to backdate a transaction into a prior
  month — `seedTrend`/`spendOn` in `e2e/a11y.spec.ts` are the reusable pattern.
- **Bundle:** 120 KB gz ceiling, 109.90 KB now (~10.1 KB headroom). Hand-rolled SVG only — **no
  charting library.**
- **Known carries (not this slice's regressions):** `recurring.test.ts`'s post-due occurrence-count
  case, and a flaky `getByLabel("Account")` collision in the forecast e2e test — both reproduce on
  `main` via `git stash`, see §4.

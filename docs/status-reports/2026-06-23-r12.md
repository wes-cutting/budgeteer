## Challenge findings (before building)

The kickoff asked to add a recoverable error state without changing the API, and to **name what the
boundary actually protects** + decide top-level-only vs. also per-analysis-view. I read `main.tsx`,
`App.tsx`, `AnalysisSection.tsx`, and the existing fetch-error handling, and confirmed:

- **Riskiest assumption — what the boundary actually catches.** React error boundaries catch errors
  thrown during the **render / lifecycle of descendants only** — **not** event handlers, **not**
  async/promise rejections, **not** errors in their own render. Async/fetch errors *already* degrade
  gracefully today: every view catches its fetch into local `loadError`, and the Dashboard tracks
  `loadError`. So this boundary's job is narrow and real: a **render-time throw** (a malformed shape
  that throws mid-`.map()`, an undefined access in a derived-data calc) — which, because `main.tsx`
  renders `<App/>` directly, currently unmounts the whole tree to a blank white screen. That, and only
  that, is what R12 fixes. The boundary is render-only **by design**, not by omission.
- **Decision — one component, two placements.** Top-level wrapping `<App>` is the must-have (never
  blank-screen the whole app). I **also** wrapped the active view inside `AnalysisSection` with
  `key={tab}`: the analysis views do the most derived-data render work (the likeliest throw site) and
  already sit behind a persistent `<nav>` that mounts one view at a time, so a localized fallback keeps
  the sub-nav usable and `key={tab}` auto-resets the boundary on every tab switch. The other
  single-view routes (account register, envelope ledger, needs, templates, recurring) are covered by
  the top-level boundary — adding a boundary to each would be over-engineering for the same protection.
- **Regression net.** A `ErrorBoundary.test.tsx` unit test is the generic proof (child throws → an
  accessible fallback with role/heading/button; the recovery button clears the error, runs `onReset`,
  and re-renders the recovered child). For the **real** top-level boundary + the **a11y of the new
  fallback**, a **dev-only `?boom` hook** in `main.tsx` (`import.meta.env.DEV`-gated) forces a render
  crash, so `e2e/error-boundary.spec.ts` and a new `e2e/a11y.spec.ts` case drive the live app to the
  fallback and axe-scan it. The hook is constant-folded to `false` by `vite build` and tree-shaken from
  the production bundle (verified: 0 occurrences of the crash string / `boom` in `dist`), so it adds no
  production fault surface — the gate's build step is the proof.
- **a11y watch-item (caught live).** The fallback is new user-facing UI. The e2e axe scan passed, but
  Playwright defaults to **light** mode; a live preview under `prefers-color-scheme: dark` showed the
  default-color fallback rendering **black-on-dark** (unreadable) because the app sets no colors and the
  fallback inherited them. Fixed by pinning the card's **own** high-contrast colors + `colorScheme:
  light` (so the native button keeps dark-on-light contrast on the white card). The fallback now reads
  in either scheme.

**Decision:** one `FastifyPluginAsync`-free `ErrorBoundary` class component, two placements
(top-level in `main.tsx`; per-tab in `AnalysisSection` via `key={tab}`), a self-contained accessible
high-contrast fallback, a unit test for boundary behavior, and a dev-only `?boom` hook driving the e2e
+ a11y coverage. No API/schema/domain change.

## What R12 built

### New: `apps/web/src/ErrorBoundary.tsx`

A `Component<Props, State>` with `getDerivedStateFromError` (sets `error`) + `componentDidCatch`
(logs to the browser console for diagnosis — structured server logging is R13's concern). Props:
`children`, an optional `resetLabel` (default `"Reload"`), and an optional `onReset` (default = a
full-page reload). The recovery button clears the error state then runs `onReset`. The fallback:

```
<main>
  <div role="alert" style={PANEL}>   // PANEL = #1a1a1a on #fff, 2px #991b1b border, colorScheme:light
    <h1>Something went wrong</h1>
    <p>… Your data is safe — nothing was changed.</p>
    <button>{resetLabel}</button>
  </div>
</main>
```

### Changed: `apps/web/src/main.tsx`

Wraps `<App>` in `<ErrorBoundary>` (top-level). Adds a **dev-only** render-crash hook:
`import.meta.env.DEV && new URLSearchParams(location.search).has("boom")` renders a throwing
`CrashOnRender` inside the boundary instead of `<App>`. `import.meta.env.DEV` is `false` in the
production build, so the branch (and `CrashOnRender`) are dead-code-eliminated.

### Changed: `apps/web/src/AnalysisSection.tsx`

Wraps the active-view block in `<ErrorBoundary key={tab} resetLabel="← Dashboard" onReset={onBack}>`.
`key={tab}` makes switching tabs (via the still-mounted `<nav>`) mount a fresh boundary, so a crashed
view never sticks across tabs. The six views and the sub-nav are otherwise unchanged.

No other source file changed. **No `api.ts` / `fakeApi.ts` / `06_API_CONTRACT.md` / `05_DATA_MODEL.md`
/ `04_DOMAIN_MODEL.md` change.**

## Commands & gotchas (cold-start)

Gate: `npm run typecheck && npm run lint && npm test && npm run test:e2e && npm run format && npm run build --workspace @budgeteer/web && npm audit --omit=dev --audit-level=critical`

- **SCA note:** `npm audit` reports 3 **high**-severity advisories (kysely chain); the gate is
  `--audit-level=critical`, so it passes (exit 0). Pre-existing, not introduced by R12.
- **Error boundaries are render-only.** They do **not** catch event-handler or async/fetch errors —
  those already surface via each view's `loadError`. Don't reach for the boundary to handle a failed
  fetch; that path is separate and already graceful.
- **The `?boom` hook is dev-only.** Visit `http://localhost:5173/?boom=1` (Vite dev) to see the
  top-level fallback. It is gated on `import.meta.env.DEV` and tree-shaken from `vite build` — never
  rely on it in production.
- **`key={tab}` is load-bearing in `AnalysisSection`.** Without it, a boundary that caught an error in
  one tab would keep showing the fallback after switching tabs (error state is sticky until reset).
  The key gives each tab a fresh boundary.
- **The app sets no global colors** — a fallback (or any new surface) that relies on default text/bg
  colors is **black-on-dark** under `prefers-color-scheme: dark`. Pin explicit high-contrast colors on
  any emergency/standalone surface (the fallback does: `#1a1a1a`/`#fff` + `colorScheme: light`).
- `reuseExistingServer: !process.env.CI` — kill stale `:3001`/`:5173` before e2e if you ran the preview.

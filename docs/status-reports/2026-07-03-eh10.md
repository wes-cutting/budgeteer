## 4. Notes / decisions

| Item | Note |
| ---- | ---- |
| **Restore refuses, never merges** | A store with any user data (beyond the untouched seed household) is refused naming the occupied tables. Merging two ledgers is not recovery; the flow is `db:reset` → restore. Same never-delete-financial-data policy as migration `0002` |
| **Schema stamp is additive** | `version` stays 1; `schema.migrations` is optional on read. Newer-schema files are refused naming the missing migration; **older (pre-stamping) files restore with a warning** — they're the owner's real accumulated backups, the very files this item protects |
| **PG is the row validator** | The zod schema covers the envelope only; re-typing every column would duplicate the schema (EH1's lesson). One documented loose seam (`LooseDb`, no `any`); constraints + one transaction give fail-loud, all-or-nothing semantics. A historical `0002`-violating duplicate in an old file fails the transaction naming the conflict — manual resolution, consistent with EH14 |
| **CLI-only is a security decision** | `GET /export` has no auth (V1, loopback-bound per EH11); a restore endpoint would be a remote wipe-and-replace primitive. Recorded in `06_API_CONTRACT`; revisit with `#19` |
| **Equivalence is defined at the export representation** | `timestamptz` truncates to JS ms at export; `date` columns serialize as UTC-midnight ISO (SPIKE-09 F4). What the export captures is exactly what restore reproduces |
| **Perf/flake watch items** | `perf.test.ts` fine; no e2e click-stability timeouts. Both stay watch-only |

## 6. Commands & gotchas (cold-start)

```sh
npm run typecheck && npm run lint && npm run format
npm test                 # 389 passing
npm run test:e2e         # 94 Playwright — STOP any dev servers on :3001/:5173 first (K20)
npm run build --workspace @budgeteer/web   # 117.32 KB gz

# The recovery drill (from apps/api):
npm run db:reset && npm run db:restore -- path/to/budgeteer-backup-YYYY-MM-DD.json
```

- **Restore needs an empty store** — it refuses occupied ones by design (`db:reset` first, or a
  fresh `PGLITE_DIR`).
- **Schema changes are new migration files** (`0003-…` + registry + `05_DATA_MODEL`, same
  change); committed files including the baseline are FROZEN.
- **e2e needs to own the stack** (K20): stop dev servers on `:3001`/`:5173` before
  `npm run test:e2e`, restart them after.
- Client calendar dates via `apps/web/src/dates.ts` (never `toISOString()`); API tests pass
  absolute dates explicitly; the server requires every user-facing date/month param (EH8).
- `kysely_migration` is Kysely-managed, not in the `DB` type — raw `sql` only
  (`backupService`/`restoreService` do this at runtime now, not just tests).

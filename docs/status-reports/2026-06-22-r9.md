## Challenge findings (before building)

The kickoff posed one design question: **net the accounts into one line, or split assets vs.
liabilities vs. net?** Resolved by reading `analysisService.envelopeSpend` (#11's monthly grid),
`creditUtilization`/`debtPayoff` (#14a/#14b's `owed = −balance` trend), and `v_account_balances`:

- **The sign convention makes this almost free, and de-risks the whole slice.** A liability
  account already stores its debt as a **negative balance**, so `Σ (signed balances of all accounts)
  = assets − what-you-owe = net worth`, with **no special-casing**. The headline net-worth value is
  unambiguous regardless of any classification debate. Account↔account transfer legs (`kind='transfer'`)
  sum to **zero** across the two accounts (correctly included, self-canceling); envelope reallocations
  never touch `accounts`, so they're irrelevant.
- **Decision: split into Assets / Liabilities / Net — classified by account *kind*, not by sign.**
  Strictly more useful for "how am I doing overall?" (is debt shrinking *or* are assets growing?) at
  no extra query cost, and it keeps the invariant **`net = assets + liabilities`** (liabilities carried
  signed/negative). Rejected a **sign-based** split (`Σ balance>0` vs `Σ balance<0` — the approach the
  future `R4` row currently sketches) because over a **trend** an account would flip columns as its
  balance crosses zero (an overdrawn checking account jumping to "liabilities" for a month), making the
  lines jagged and misleading. **By-kind keeps each account in a stable column.**
- **E2E reality.** Net worth is a **household-wide aggregate with no per-entity row**, and the e2e DB
  accretes across parallel spec files — so an absolute dollar figure can't be asserted (every other
  test's accounts pollute the sum). The e2e instead asserts the **invariant `net = assets +
  liabilities`** off the rendered current totals (true at any render, immune to pollution) plus the
  current-month trend row's presence — a genuine data→API→domain→UI check. Exact values are covered by
  the unit + integration tests.

**Decision:** a read-only household aggregate → pure-domain `netWorthOverTime` → thin view. No schema
change, no new guard, no new domain primitive (only a tiny `isLiabilityKind` classifier).

## What R9 built

### `isLiabilityKind` + `netWorthOverTime` (pure domain)

`packages/domain/src/account.ts` gained `LIABILITY_KINDS` (`['credit','loan']`) + `isLiabilityKind`.
New `packages/domain/src/networth.ts`: `netWorthOverTime(flows)` buckets per-(period, kind) net flows
into asset/liability sides, sorts periods ascending, and cumulates each side into a period-end
`{ assetsCents, liabilitiesCents, netCents }` trend (one point per period **with activity**; the
headline = the final point, or 0/0/0 when empty). No I/O, no `Date.now` — unit-testable in isolation.

### `analysisService.netWorth(grain)` + route

The service gathers per-(period, kind) net flows with `sum(t.amount_cents)` over **all** transactions
joined to `accounts` (no kind/transfer exclusions), grouped by `to_char(occurred_on, fmt)` and `kind`,
then runs the pure function and tags on `grain`. Route `GET /analysis/net-worth?grain=month|year`
(reuses the existing `SpendQuery` generic; default `month`; malformed grain → `400`).

### Frontend

**`api.ts`:** `NetWorthPoint`/`NetWorthReport` types + `getNetWorth(grain)` (interface + `httpApi`).
**`fakeApi.ts`:** `getNetWorth` mirrors the server — groups `txns` (which already carry opening rows
and both transfer legs) by (period, kind) and runs the **same** `netWorthOverTime`.
**`NetWorthView.tsx`:** a grain toggle (Monthly/Annual), a "Current totals" table (Assets /
Liabilities / Net), and an over-time trend table (period × Assets/Liabilities/Net), with
loading/empty/error states. Wired into `App` + a "Net worth" header button in `Dashboard`.

Gate: `npm run typecheck && npm run lint && npm test && npm run test:e2e && npm run format && npm run build --workspace @budgeteer/web && npm audit --omit=dev --audit-level=critical`

- **SCA note:** `npm audit` reports 3 **high**-severity advisories; the gate is
  `--audit-level=critical`, so it passes (exit 0). Pre-existing, not introduced by R9.
- **Aggregate-view e2e:** net worth is household-wide with **no per-entity row**, and the shared e2e
  DB accretes across parallel specs — assert the **invariant** (`net = assets + liabilities`) and
  structural presence, never an absolute dollar figure.
- **Opening rows count toward net worth.** Unlike `#11` (which reads `allocations`, so the
  allocation-less opening row is invisible), net worth sums **all** transactions — the opening row is
  included (it *is* the starting balance). A `$0` opening row is a harmless 0-flow.
- `reuseExistingServer: !process.env.CI` — kill stale `:3001`/`:5173` before e2e if you changed
  server code.

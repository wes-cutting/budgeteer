**Resume here:** The **third UX12 thread** is done. UX12 bundles **four independent polish threads**
(skeletons · toasts · destructive-action confirms · inline validation); threads 1 (destructive
confirms, UX12a) and 2 (skeleton loaders, UX12b) shipped earlier. This slice ships **thread 3 —
inline validation**: money-amount inputs that failed **silently** (the quick-add / register amount
just greys out `AllocationEditor`'s Save) or only at the **server** round-trip (transfer · move-money
· add-account starting-balance) now surface a **field-level** message the instant an amount can't
parse. A new pure **`amountFieldError`** (`apps/web/src/validation.ts`, reusing the single penny-exact
parser `tryParseMoney`) drives a new **`FieldError`** primitive (`ui/Feedback.tsx`): the input gets
`aria-invalid` + `aria-describedby` and a short, format-showing message ("Enter an amount like
12.34.") — the **text** carries the state, **never colour** (`--color-danger` ~6.5:1 light+dark). The
three submit-driven forms also **guard the mutation** client-side; semantic rules (positive amount,
funds, distinct envelopes) stay **server-authoritative**. **No ADR/spike, no new dependency** — additive
UI reusing an existing pure parser + design-system primitives (§11 fast-path). Gate green: **349 Vitest
passing** (+6; 1 pre-existing unrelated carry — see §4) **+ 86 e2e** (+3), build **111.68 KB gz** (+0.24
KB; < 120 KB, ~8.3 KB headroom). **Next:** the last `UX12` thread — success toasts (`UX12c`, needs a
new Radix Toast primitive) — then `UX13`–`UX15`.

**Resume here:** **`#14a` is `Done` and gate-green** — the **fourth analysis slice** and the **first
half of the split `#14`**. The owner split the roadmap's "debt & credit trends" (as `#7` was split):
**credit utilization now (`#14a`)**, installment-debt **payoff-% later (`#14b`)**. For every
`kind='credit'` account Budgeteer now shows **owed vs. its credit limit** — current utilization
(`owed ÷ limit`), a **month-by-month utilization trend**, and a **portfolio roll-up** across cards —
where **owed = −balance** (a credit account's derived balance is ≤ 0 when in debt; the **stored ledger
sign is unchanged**, only the analysis read flips it). **No spike** was warranted (challenged below):
like `#11`/`#12` it aggregates existing balances + one stored number — no forward projection (`#13`
was the area's one genuine unknown), no external source. Built data → API → UI: a **pure domain**
`creditUtilization` → read-only `analysisService.creditUtilization` → `GET /analysis/credit-utilization`
+ `PUT`/`DELETE /accounts/:id/credit-limit` → a thin `CreditView`. **One new table** (`credit_limits`,
the stored limit); owed/utilization/trend/roll-up are **derived**. **Next: `#14b` (debt payoff %) — or,
if the owner prefers to defer the meatier loan-modeling to post-V1, `#15` (backup/export, first

## What #14a did

**Challenged the scope FIRST, then settled the modeling with the owner.** The roadmap flagged `#14`
as a "larger area — likely spawns its own feature specs," so scope was put to the owner before any
build (FEAT-014a §11):

- **What V1 covers → credit utilization (not payoff-%); split into `#14a`/`#14b`.** "Debt & credit
  trends" bundles two different things with **no shared reference number**: utilization (`owed/limit`,
  fits the existing `kind='credit'`) and payoff-% (`1 − owed/original`, needs an original-principal /
  loan model — there is **no `kind='loan'`**). Utilization needs exactly **one** new reversible number
  and maps onto what the schema already models; payoff-% is the meatier follow-up → **`#14b`**.
- **Grain → per-account + a portfolio roll-up.** Per-account is the unit; the roll-up
  (`Σ owed ÷ Σ limit` over cards with a limit) is cheap and matches the "credit health at a glance"
  framing.
- **The limit store → a separate `credit_limits` config table** (mirrors `envelope_targets`), **not**
  a column on `accounts`. Keeps `accounts` lean, matches the FEAT-012 pattern, and leaves room for
  `#14b` to add original-principal alongside. A **reversible, non-effective-dated** field ⇒ a
  **definition in the spec, no ADR**.
- **Sign convention (resolved an open Q) → owed = −balance at the analysis layer.** The
  `04_DOMAIN_MODEL` open question "liability (credit-card) balance sign convention" is now resolved:
  the **stored** balance stays the signed sum of transactions; only the **analysis** read interprets a
  credit account's balance as `owed = −balance`.

**No spike — said so explicitly (the kit's "challenge the plan" gate).** `#13` was the analysis area's
one genuine unknown (a forward projection). `#14a` **aggregates** existing derived balances + one
stored number — no projection, no external/legacy source. The single piece of new logic — a
**cumulative** balance per period (vs. `#11`'s per-period **net flow**) for the trend — is a known
technique, **unit-proven** against fixed fixtures, not a spike-level unknown.

**The build (vertical):**

- **Domain (pure)** — new [`credit.ts`](../../packages/domain/src/credit.ts): `creditUtilization`
  (per-account owed/available/utilization + cumulative monthly **trend** + portfolio **roll-up**) over
  `utilizationBps` (`round(max(0, owed)/limit × 10000)` — floored at 0, **not** clamped above 100%).
  No I/O. Exported from the domain index.
- **API (read + a tiny store)** — new
  [`creditLimitService`](../../apps/api/src/services/creditLimitService.ts) (`set`/`clear`, restricted
  to `kind='credit'` → `ValidationError`/`400`) mirroring `targetService`; new
  [`analysisService.creditUtilization`](../../apps/api/src/services/analysisService.ts) gathers the
  credit accounts, their derived balances (`v_account_balances`), their limits, and a per-account
  monthly net-flow query, then calls the pure domain function. Routes
  `GET /analysis/credit-utilization`, `PUT`/`DELETE /accounts/:id/credit-limit`. **One new table**
  (`credit_limits`, idempotent migration); **no balance-view change**.
- **UI (thin)** — new [`CreditView`](../../apps/web/src/CreditView.tsx): a roll-up `<dl role="status">`,
  a per-account `<table>` with an inline **limit** editor (Save/Clear), and per-account
  **trend** tables. Reached from a Dashboard **Credit** button. Every ratio is **text** (percentage,
  "over limit", "— (set a limit)", "$… credit"); money via web `formatCents`/`formatMoney`, percentage
  via a new `formatBps`; never touches the datastore.

## Commands & gotchas (cold-start)

Gate is unchanged: `npm run typecheck`, `npm run lint`, `npm test`, `npm run test:e2e`, `npm run format`, `npm run build --workspace @budgeteer/web`.

- `npm run format` is `prettier --check .` — it checks source `.ts`/`.tsx` but **ignores `docs/**` and
  `*.md`**, so run `npx prettier --write .` before the gate (new **source** files trip it; new docs don't).
- `npm run lint` is a **zero-warning** gate; unused test bindings fail it (caught two here — assign only
  what you use).
- `npm run test:e2e` needs Chromium once (`npx playwright install chromium`); it boots both servers
  (web on `:5173` to match the API CORS allowlist) and the journey now ends with a **Credit** step.
- The `.claude/launch.json` `api` server is `npm run start` (**no watch**) — restart it to pick up API
  changes during local preview. The Credit view needs **both** servers.
- **Owed sign:** a credit account in debt has a **negative** derived balance; `owed = −balance`. To
  seed a card in debt in tests, open it with a **negative** starting balance (e.g. `"-1500.00"`) or
  post dated withdrawals. The trend's final cumulative **must** equal the current owed — an asserted
  invariant; keep it that way.
- CI (`.github/workflows/gate.yml`) stays **manual-only** (`workflow_dispatch`) — unchanged.

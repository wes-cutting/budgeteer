**Resume here:** Accounts can now be **reconciled to the bank**. On an account's register, the
**Reconcile** panel shows Budgeteer's derived balance, takes your real bank balance, shows the
**live signed difference** (`bank − Budgeteer`), and **records** the reconciliation — snapshotting
the derived balance + statement + date so each account keeps a **"last reconciled"** history. V1
is a **plain compare** (owner decision): no per-transaction *cleared* concept, and on a mismatch
it just **shows** the difference (no auto-adjustment). A reconciliation is a recorded compare — it
creates no transaction and changes no balance. Resolves the PRD §9 reconcile open question
(FEAT-010). **Also fixed this session (not a slice):** the browser app couldn't reach the API
(missing CORS → "Failed to fetch") — added a `@fastify/cors` allowlist (`CORS_ORIGINS`) + repo-root
`.env` auto-load. **Next: the analysis area (`#11`–`#14`)** — the core loop + transfers + recurring
now produce real data to analyze.

## Commands & gotchas (cold-start)

Same as #01–#09. New: `POST /accounts/:id/reconciliations` `{ statementBalance, reconciledOn? }`
+ `GET` the same (history newest-first). `statementBalance` may be **negative** (uses
`parseMoney`, not positive-magnitude). A reconciliation snapshots `derived_balance_cents` and
never mutates balances; `difference = statement − derived` is derived. **CORS:** the API needs
`CORS_ORIGINS` (allowlist; dev default = Vite origin) or the browser gets "Failed to fetch"; the
repo-root `.env` now auto-loads (dotenv for API, Vite `envDir` for web).

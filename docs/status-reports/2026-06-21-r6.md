## Challenge findings (before building)

Read `docs/05_DATA_MODEL.md` and `migrate.ts` before writing a line of code.

**Key cascade chain discovered:**
- `allocations.transaction_id → transactions` **ON DELETE CASCADE** → deleting a transaction auto-removes its splits (no app code needed)
- `transactions.transfer_id → transfers` **ON DELETE CASCADE** → deleting the parent `transfers` row auto-removes both legs (which then cascade their allocations)

**Decision: hard delete (not soft)**
`kind='transfer'` legs cannot be individually deleted — they must be torn down as a pair
(otherwise one account gains or loses money). The endpoint rejects transfer legs with 409
Conflict and routes callers to `DELETE /transfers/:id`.

**Opening balance transactions** (`kind='opening'`): allowed to delete. The unique
constraint on `(account_id) WHERE kind='opening'` simply means "at most one," so deleting
it is valid. No special guard needed.

## What R6 built

### New error class: `ConflictError` → 409

Added `ConflictError` to `apps/api/src/services/errors.ts`. Used when a transfer leg is
targeted via the wrong endpoint ("Transfer legs must be deleted as a pair — use DELETE
/transfers/:id."). Maps to 409 in the route handler.

### `transactionService.remove(id)` (service layer)

Fetch the transaction (household-scoped) → 404 if missing → 409 if `kind='transfer'` →
hard delete (allocations cascade via FK).

### `transferService.remove(id)` (service layer)

Fetch the transfer (household-scoped) → 404 if missing → hard delete (both transaction
legs cascade via FK; their allocations cascade further).

### Two new routes in `server.ts`

- `DELETE /transactions/:id` — 204 on success, 404 on missing, 409 on transfer-leg
- `DELETE /transfers/:id` — 204 on success, 404 on missing

### Frontend

**`apps/web/src/api.ts`:** added `deleteTransaction(id)` and `deleteTransfer(id)` to the
`Api` interface and `httpApi` implementation.

**`apps/web/src/test/fakeApi.ts`:** implemented `deleteTransaction` (splice the txn) and
`deleteTransfer` (splice both legs by `transferId`).

**`apps/web/src/AccountRegister.tsx`:**
- `deletingId` state tracks the in-flight delete
- `deleteRow(txn)` — calls `api.deleteTransfer(txn.transferId!)` for transfer legs, or
  `api.deleteTransaction(txn.id)` for normal/opening rows; calls `load()` on success
- Delete button on every transfer row (`aria-label="Delete transfer"`) and every
  normal/opening row (`aria-label="Delete transaction"`)

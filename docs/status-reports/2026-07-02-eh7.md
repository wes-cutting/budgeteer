**Resume here:** `EH7` is **done, gate-green — and the gate is now FULLY green for the first time
since ~June 22** (367/367 Vitest, no carried failure; 94/94 e2e). The API no longer reaches for the
ambient wall clock below the composition root: `buildServer` accepts an injected **`clock?: Clock`**
(`() => Date`, default = the real clock) and threads it to the three services that derive "today"
(`accountService` opening-balance date, `recurringService` due-ness ×3, `analysisService` forecast
start) and to every route plugin (`RouteOptions.clock`) for the five route-level defaults (register
month window, transaction/transfer/reconcile default dates, analysis default month, backup
filename). `todayStr`/`currentMonthRange` now **require** the clock parameter, so a new call site
cannot silently regress to calendar-dependent behavior. **Root cause of the carried recurring
failure, finally diagnosed:** the post-due test's 4 generated occurrences all posted correctly — but
the register endpoint **defaults its window to the server's current month** (R8), so on any real
calendar day ≤ the 21st the −21/−14/−7 relative-date fixtures fell in the *previous* month and the
register showed 1 row, not 4. The test could only pass on days 22–31 — a perfect specimen of the
EH7/EH8 diagnosis. Fixed properly: the suite pins `createTestApp({ today: "2026-03-28" })` and uses
**absolute fixture dates**. The **flaky forecast-e2e selector** (UX10 §4) is also fixed — `exact:
true` on `getByLabel("Account")`, the established R1 convention (the Forecast chart's `role="img"`
aria-label can contain the substring "Account"). **No API-contract/schema/domain/web-app change.**
**Next:** the remaining review items `EH8`–`EH14` + the sheet-parity features `S7`/`S9` — order is
the owner's call (the review suggests `EH11` next: the one-line `HOST` default).

## 4. Notes / decisions

| Item | Note |
| ---- | ---- |
| **Root cause ≠ the domain** | `dueOccurrences` was always correct (posted = 4, balances right). The failure was the **register's server-derived default month window** clipping prior-month occurrences — the test asserted through a second, calendar-dependent seam. Diagnosis matters: patching the fixture with wider relative offsets would have "fixed" it 10 days a month |
| **Required-parameter clocks** | `todayStr(clock)`/`currentMonthRange(clock)` deliberately have **no default**; only `buildServer` (and the service factories' signature defaults, which exist for direct-construction convenience) reference `systemClock`. The compiler now enforces the "clock is passed in" rule at every new call site |
| **Scope: calendar derivation only** | `archived_at`/`updated_at`/`exportedAt` write-timestamps still use `new Date()` — they are *event instants*, not calendar-date derivations, drive no assertions, and threading them adds churn without payoff. Recorded here so it's a decision, not an oversight |
| **EH8 unaffected** | The injected clock makes "today" *deterministic*, not *correct-for-timezone* — `todayStr` still derives UTC. The timezone-policy decision (EH8) now has its seam ready: it becomes a change to how the clock's instant is converted, in exactly one place |

  absolute fixture dates. Never use relative-date fixtures against the real calendar (this slice
  exists because one did).

  either pin the clock so fixtures land in-month, or pass an explicit `?from=&to=`.

  factory parameter (services); only composition roots may touch `systemClock`.

## What the linter caught (and how it was fixed)

- **`react-hooks/rules-of-hooks` error in `AllocationEditor.tsx`:** a plain helper named
  `useRemaining` (fill a row with the remaining amount) — the `use` prefix is reserved for hooks, so
  the plugin treated the `onClick={() => useRemaining(i)}` call as a hook called inside a callback.
  **Fixed properly by renaming** `useRemaining` → `fillRemaining` (it was never a hook). Behaviour
  and the user-facing "use remaining" button label are unchanged.
- **Two unsuppressed `exhaustive-deps` warnings** (`NeedsAllocation`, `TemplatesView`) for the
  mount-load `useEffect(() => void load(), [])` pattern. The codebase's established decision (already
  in `ReconcilePanel`, `RecurringView`, `AccountRegister`) is an explicit `eslint-disable-next-line`
  for these intentional run-once effects; added the same to the two stragglers so the pattern is
  consistent and the zero-warning gate holds. (Switching the pattern to `useCallback` everywhere is a
  larger change, out of EH4's "wire up the linter" scope.)

## Design notes / decisions

- **Flat config, ESLint 9.** The current standard; co-locates ignores (`dist`, `node_modules`,
  `spikes/`) and per-area overrides (node globals for api/domain, browser globals + react-hooks for
  web) in one typed file via `tseslint.config(...)`.
- **`recommended`, not `recommendedTypeChecked`.** Type-aware linting would duplicate work `tsc`
  already does strictly and would need tsconfig wiring per workspace. Kept lint syntactic and fast;
  type-checked rules remain a possible future enhancement (note for `#16` hardening).
- **react-hooks rules referenced explicitly** (`rules-of-hooks` / `exhaustive-deps`) rather than via
  the plugin's preset, so the config is independent of the plugin's evolving flat-config export
  shape.
- **Zero-warning gate (`--max-warnings 0`).** Matches the project's strict gate philosophy and makes
  every suppression a deliberate, reviewed choice; a new unsuppressed warning fails the gate.

**Resume here:** Accounts can now **transfer money between each other** (double-entry,
[ADR-0004](../adr/ADR-0004-transfer-modeling.md), de-risked by
[SPIKE-04](../spikes/04-transfer-modeling.md)). A transfer is a `transfers` parent + **two
linked `kind:'transfer'` transactions** (`−X` source / `+X` destination): account balances
re-derive and stay **conserved**, the legs are **excluded from needs-allocation** (relocated
money is already budgeted), and each leg shows in the register labeled **"Transfer to/from
&lt;account&gt;"** with no allocation editor. Owner decisions this block: **split `#7` into
`#7a`/`#7b`**, and **negative balances allowed**. **Next: `#7b` envelope↔envelope reallocation**
— the second ADR-0004 primitive (a dedicated `envelope_transfers` table + the
`v_envelope_balances` two-source change + its own UX) — then refunds `#8`, recurring `#9`, and
the analysis area (`#11`–`#14`).

## Commands & gotchas (cold-start)

Same as #01–#05. New: `POST /transfers` `{ fromAccountId, toAccountId, amount, occurredOn?,
memo? }` → `201 { transfer }`. Transfer legs are `kind:"transfer"` with `transferId` +
`transferCounterpartName`; they appear in the account **register** but **not** in
needs-allocation. `v_account_balances` needed **no** change (already sums all kinds). Schema
evolution is forward-only/idempotent in `apps/api/src/db/migrate.ts` (kind check renamed to
`transactions_kind_chk`).

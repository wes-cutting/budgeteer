**Resume here:** Budgeteer now has **recurring transactions**. A **rule** captures an account +
direction + fixed amount + payee/memo + a **split** + a **schedule** (weekly / biweekly /
monthly, anchored on a date). A **Post due** action generates the concrete transactions due
on/before today and advances each rule's `next_occurrence_on` **cursor** — **idempotent** (re-run
posts nothing; no background worker). Generated rows are ordinary transactions (register +
balances; a partial split lands in **needs-allocation**) tagged with `recurring_id`; deleting a
rule keeps them (FK set null). The schedule math is a pure domain core (`recurring.ts`) with the
monthly **anchor-day clamp** (31 → Feb 28/29 → 31). **Next: reconcile `#10`**, then the analysis
area (`#11`–`#14`).

## Commands & gotchas (cold-start)

Same as #01–#08. New: `GET/POST/DELETE /recurring` + `POST /recurring/post-due` (bodyless). A
rule stores a **positive** `amount_cents` + `direction`; generation applies the sign and the
per-line `refund` flag. **Idempotency** is the advancing `next_occurrence_on` cursor (persisted),
not a uniqueness constraint. `dueCount`/Post-due use the server's `new Date()` "today"; tests
anchor relative to `Date.now()` so they're calendar-independent. Generated txns carry
`recurringId`; balance views are unchanged.

**Resume here:** **`#13` is `Done` and gate-green** — the third analysis slice and the area's only
**forward-looking** one. Budgeteer now **projects** an account's running **cash balance** over a
horizon (default 90 days), starting from the derived balance (`v_account_balances`) and applying
**future scheduled recurring events** (FEAT-009 — the firm core) plus, as a toggle (default **on**),
**expected discretionary spend** from monthly **targets** (FEAT-012), netted so already-scheduled or
already-spent money is never double-counted. It surfaces the headline answers — **ending balance**,
**minimum balance + date**, and **first-negative date** ("will I make it to payday?"). De-risked by
a real throwaway spike ([SPIKE-05](../spikes/05-cashflow-forecast.md)) — the owner asked for it once
they chose the richer "+ targets" model, which is exactly what introduced the one fuzzy fork (placing
a monthly target on a dated cash timeline without double-counting). Built data → API → UI: a **pure
domain** `cashFlowForecast` (+ `runningBalance`/`scheduledEvents`/`expectedSpendEvents`) → a read-only
`analysisService.cashFlowForecast` that gathers the inputs → `GET /analysis/cash-flow-forecast` → a
thin `ForecastView` (account picker + horizon + expected-spend toggle + summary + running-balance
table). **No schema change** (a pure projection, like `#11`). **Next: `#14` — debt & credit trends.**
The kickoff prompt at the bottom frames it.

## What #13 did

**The modeling questions (challenged first, then settled with the owner + a spike).** A forecast
*projects* the future, so unlike `#11`/`#12` it carried a genuine modeling unknown. The forks were put
to the owner before building:

- **What is forecast → per-account cash balance.** The "will I go negative before payday?" question.
  Recurring rules and balances are account-scoped, so per-account is the firm unit. Envelope-balance
  forecasting and multi-account/total cash are documented post-V1.
- **Which inputs → scheduled recurring events + expected spend from targets (both).** Scheduled is
  the concrete, dated core; the owner chose to **also** fold in monthly targets as expected
  discretionary spend (the richer "budget projection"). That choice reintroduced the fuzzy fork → a
  spike was warranted.
- **Horizon & grain → event-stepped running balance, 90-day default** (caller-set, capped `[7,365]`).
  Exact, no bucketing; paychecks appear as the up-steps (the pay-period frame falls out for free).
- **The fuzzy fork, resolved by [SPIKE-05](../spikes/05-cashflow-forecast.md):** expected spend =
  per month `Σ max(0, target − actualThisMonth − scheduledThisMonth)` (current month un-prorated,
  future-tail prorated), spread **even-daily**, attributed to the forecast account. The netting is the
  **anti-double-count** rule — fully-scheduled/already-spent envelopes (Rent, Utilities) contribute
  **$0**. Owner also confirmed: the firm **scheduled-only** line is a toggle (default **on** = +expected).

**The spike (warranted, time-boxed, throwaway).** [SPIKE-05](../spikes/05-cashflow-forecast.md)
(8/8 under strict `tsc`) proved: the recurring engine projects the future **unchanged** when fed the
horizon as its bound; the event-stepping core is exact (ending/min/first-negative); and — the gold —
naively adding raw targets would double-count **$1,620/mo** on the realistic fixture, while the netted
residual is the correct **$550/mo** discretionary. The spike's code is disposable (its findings live
in the feature spec + this slice + the report); a hygiene follow-up can discard `spikes/05-...` (as
`EH6` did for spike-04).

**The build (vertical):**

- **Domain (pure)** — new [`forecast.ts`](../../packages/domain/src/forecast.ts): `cashFlowForecast`
  (assembly) over `runningBalance` (event-stepping; conservative same-day order — most-negative first),
  `scheduledEvents` (engine fed the horizon, `date > today`), `expectedSpendEvents` (the netted/spread
  residual). Reuses `dueOccurrences`/`anchorDayOf` (FEAT-009, **unchanged**) + `splitEvenly`/`daysInMonth`.
  No I/O; `today` passed in. Exported from the domain index.
- **API (read-only)** — new [`analysisService.cashFlowForecast`](../../apps/api/src/services/analysisService.ts):
  gathers the derived balance, the account's recurring rules + lines, the household targets, and the
  current-month outflow actuals (the FEAT-012 basis), then calls the pure domain function. Route
  `GET /analysis/cash-flow-forecast?accountId&horizonDays=90&includeExpected=true` → `200 { forecast }`
  (bad horizon → `400`; missing account → `404`). **No new table or view.**
- **UI (thin)** — new [`ForecastView`](../../apps/web/src/ForecastView.tsx): account `<select>`,
  horizon (30/60/90) buttons, **Include expected spend** checkbox (default on), a `role="status"`
  summary (starting / projected ending / lowest point + date / goes-negative), and a real `<table>`
  (caption, `scope`'d headers; signed amounts as text). Reached from a Dashboard **Forecast** button.
  Money via web `formatCents`; never touches the datastore.

## Commands & gotchas (cold-start)

Gate is unchanged: `npm run typecheck`, `npm run lint`, `npm test`, `npm run test:e2e`, `npm run format`, `npm run build --workspace @budgeteer/web`.

- `npm run format` is `prettier --check .` — it **does** check source `.ts`/`.tsx` but **ignores
  `docs/**` and `*.md`** (see `.prettierignore`), so new spec docs never trip the gate; new **source**
  files do, so run `npx prettier --write .` before the gate.
- The forecast is **"today"-relative** (the server's `todayStr()` drives the projection). The API/web
  tests anchor rules with a `plus(n)` helper off the current UTC date — don't hardcode dates in new
  forecast tests or they'll drift.
- `npm run test:e2e` needs Chromium once (`npx playwright install chromium`); it boots both servers
  (web on `:5173` to match the API CORS allowlist) and the journey now ends with a **Forecast** step.
- The `.claude/launch.json` `api` server is `npm run start` (**no watch**) — restart it to pick up API
  changes during local preview. The Forecast view needs **both** servers.
- CI (`.github/workflows/gate.yml`) stays **manual-only** (`workflow_dispatch`) — unchanged.
- **Throwaway:** `spikes/05-cashflow-forecast/` is disposable spike code (findings absorbed into the
  spec + this slice). A future hygiene chore can discard it (and its `.prettierignore` line), as `EH6`
  did for spike-04.

**Resume here:** Two fixes from the review:

1. **DB unique-violation → 409 (was 500).** The unique index on `(household_id,
   lower(btrim(name)))` is the *real* name guard: it backstops the in-app `nameExists` check and is
   the only guard under a concurrent insert. Its rejection previously surfaced as a generic 500. A
   new **`apps/api/src/services/dbErrors.ts`** (`isUniqueViolation` + `asDuplicateName`) translates
   Postgres/PGlite SQLSTATE `23505` into a **`DuplicateNameError`** (which the HTTP layer already
   maps to 409). The 6 name-writing methods are wrapped: account/envelope/template **create** +
   **rename/update**.
2. **Typed route params.** All **12** `req.params as { id: string } | { accountId: string }` casts
   — the only typed-assertion escape hatch in the codebase — replaced with Fastify **route
   generics** (`IdParams` / `AccountIdParams`), so `req.params` is typed without a cast.

**Boundary decision (flagged & honored):** the mapping lives in the **service layer**, not the HTTP
error handler. Mapping it centrally in `setErrorHandler` would have been fewer lines, but it would
put the *presentation* layer in the business of interpreting a *datastore* error code (`23505`) —
which conflicts with the non-negotiable boundary rule (ARCHITECTURE §3 / CLAUDE.md). Services own DB
access, so translating a DB error into a domain error there is the in-bounds choice; the HTTP layer
keeps mapping `DuplicateNameError → 409` as it already did.

make the now-inert `eslint-disable` comments meaningful (or remove them).

## Why the test exercises the DB directly (not via HTTP)

The in-app `nameExists` (normalize = `lower` + `btrim` + collapse-internal-whitespace) is a strict
**superset** of the DB key (`lower(btrim(name))`): any name the DB would reject as a duplicate, the
in-app check also rejects — first. So there is **no name** an HTTP request can send that passes the
app check but trips the index; the index only fires under a true concurrent insert. The test
therefore reaches the backstop by writing through `ctx.db` directly (around the service), which is
the honest way to prove SQLSTATE `23505` is emitted and mapped. The existing `…→ 409` HTTP tests
already cover the in-app path.

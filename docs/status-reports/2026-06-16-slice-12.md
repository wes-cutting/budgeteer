**Resume here:** **`#12` is `Done` and gate-green** — the second analysis slice. Budgeteer now stores
a per-envelope **recurring monthly budget target** (the half it lacked) and compares it, for a chosen
month, against **actual spend** — defined as the **outflow only** (`−Σ allocations on withdrawal
transactions`; funding deposits excluded, refund rows netted down), with `remaining = target − spent`.
Built data → API → UI: a new `envelope_targets` table (idempotent migration) + a `targetService`
(set/clear) + a **sibling** `analysisService.budgetVsActual(month)` read (FEAT-011's `envelopeSpend`
untouched) → `GET /analysis/budget-vs-actual` + `PUT`/`DELETE /envelopes/:id/target` → a thin
`BudgetVsActualView` (month picker + inline target editor). **A latent CORS bug was found and fixed**
(see below). **Next: `#13` — cash-flow forecast** (the most modeling-heavy analysis item; may warrant
a spike). The kickoff prompt at the bottom frames it.

## What #12 did

**The modeling questions (settled with the owner before building).** Budget-vs-actual is a
*definition* as much as code, so the two real forks were put to the owner first (challenge-the-plan):

- **Targets shape → one recurring monthly target per envelope.** A single amount (no
  effective-dating), stored in a small `envelope_targets` table (no row = no target). Because it is
  not an effective-dated/hard-to-reverse data decision, it is a **definition in the feature spec, not
  an ADR** (right-sized ceremony). Effective-dating can layer on later without breaking it.
- **Actual basis → spend only (outflow).** A budget is "planned vs. **spent**", so actual =
  `−Σ allocation.amount_cents` over allocations whose **transaction is a withdrawal** (`amount_cents <
  0`). This **excludes funding deposits** by construction and **nets refund rows** (FEAT-008) down. It
  is a **sibling** query to FEAT-011's net-flow (which returns net, not spend-only) — `envelopeSpend`
  is untouched.
- Decided in passing (definitions): **monthly** grain; `remaining = target − spent` (positive = under
  budget); rows = every **active** envelope plus any **archived** one with a target or spend that
  month; targets are **positive** magnitudes (a "$0 budget" is expressed as no target). Captured in
  [FEAT-012](../features/budget-vs-actual.md) §11.

**No spike.** Like #11, this derives from existing, already-validated data (`allocations`/
`transactions`) plus a trivial CRUD store; variance is pure subtraction — no external source,
integration, perf, or algorithmic unknown. Stated explicitly when challenging the plan.

**The build (vertical):**

- **Data** — new `envelope_targets` table (`05_DATA_MODEL` §2): `envelope_id` **unique** (one target
  per envelope), `monthly_target_cents > 0`, `updated_at` (mutable config). Idempotent migration
  (`create table if not exists` + unique index) in `db/migrate.ts`, so the dev/test PGlite path keeps
  doubling as the migrator. `EnvelopeTarget` added to `04_DOMAIN_MODEL`.
- **API** — new [`targetService`](../../apps/api/src/services/targetService.ts) (`set` upserts in a
  transaction after a household-scoped envelope check; `clear` is idempotent) + a new
  [`budgetVsActual(month)`](../../apps/api/src/services/analysisService.ts) read (one outflow-spend
  aggregate + targets + envelopes, assembled with the `target − spent` variance, integer cents narrowed
  at the boundary). Routes: `GET /analysis/budget-vs-actual?month=YYYY-MM` (default = current month;
  strict `YYYY-MM` → `400`), `PUT /envelopes/:id/target` (`{ amount }`, positive magnitude → `400`/`404`),
  `DELETE /envelopes/:id/target` (`204`, idempotent).
- **UI** — new thin [`BudgetVsActualView`](../../apps/web/src/BudgetVsActualView.tsx): a month picker
  (native `month` input, capped at the current month) + a real `<table>` (caption, `scope`'d headers,
  totals `<tfoot>`) where the target column is an **inline editor** (labelled input + Save/Clear);
  loading/empty/error/success states. Reached from a Dashboard **Budget** button. Money via web
  `formatCents`/`formatMoney` (EH1); never touches the datastore.

**The CORS fix (a surprise, handled).** The browser write path for targets is `PUT`/`DELETE`, which
surfaced a **latent, pre-existing bug**: `@fastify/cors`, configured with only an origin allowlist,
defaulted the preflight's `Access-Control-Allow-Methods` to `GET,HEAD,POST` — so the browser silently
blocked **every** cross-origin `PUT`/`PATCH`/`DELETE` (rename, edit-split, template/recurring delete —
all previously exercised only via the fake API and `inject`, never a real browser write verb; the
POST-only e2e couldn't catch it). Fixed by declaring the methods the API uses
(`GET,HEAD,POST,PUT,PATCH,DELETE`); verified via curl preflight + a real-browser inline-edit; **guarded**
by extending the e2e journey with a budget-target `PUT` round-trip. Documented in `06_API_CONTRACT` §1
and FEAT-012 §8.

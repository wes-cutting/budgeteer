## Challenge findings (before building)

The kickoff asked to **decide the structure first**: full-page views behind a tab strip in one
"Analysis" route, _or_ the views become `tabpanel`s of a single Analysis container? Plus: keep each
view's own fetch, keep tabs accessible, respect `prefers-reduced-motion`, keep the header manageable. I
read the code and settled it:

- **Full-page views behind a sub-nav — _not_ a single `tabpanel`.** Each view
  (`AnalysisView`/`BudgetVsActualView`/`ForecastView`/`CreditView`/`PayoffView`/`NetWorthView`) is a
  self-contained `<main>` with its own level-1 `<h1>` and its own `useEffect` fetch. Forcing six
  heterogeneous full pages into one `role="tabpanel"` would misrepresent the structure (one panel
  claiming to hold six `<main>`/`<h1>` pages) and require gutting all six components + their six test
  files. Instead a thin **`AnalysisSection`** wraps them and **mounts only the active view** — leaving
  the six views completely untouched.
- **Lazy mount preserves the per-view fetch.** Only the active view is rendered, so switching tabs
  unmounts the old view and mounts the new one → each view's fetch runs exactly when shown. No eager
  fetch-all-six.
- **Accessibility: semantically-correct buttons, not a `role="tablist"` widget.** The kickoff allowed
  either. For sibling full pages, a `<nav aria-label="Analysis views">` of buttons with
  `aria-current="page"` on the active one is the _more_ honest model (native Tab order; no roving
  `tabindex`/arrow-key machinery, which belongs to the single-`tabpanel` `tablist` pattern), and it
  avoids putting a `<main>` inside a `tabpanel`. It also keeps exactly **one `<main>`** on the page (the
  container is a fragment: `<nav>` + the active view's `<main>`). The axe scan now also covers the new
  nav and stays green.
- **It's 6 views, not 5.** The original R3 line predates `R9` — **Net worth** (`NetWorthView`, h1
  "Analysis — net worth over time") is unambiguously an analysis view and already lives in the
  analysis-views block of both specs. Grouping only 5 and leaving "Net worth" a stray header button
  would be incoherent, so all six are grouped.
- **`prefers-reduced-motion`: N/A.** No transition/animation is added on tab switch (instant swap), so
  there is nothing to gate. (Noted in the component doc-comment for the next person who adds one.)
- **Exit stays per-view.** Each view keeps its own `← Dashboard` button as the section exit (`onBack`);
  the sub-nav only switches among views. Two coherent levels: tabs switch analysis views; `← Dashboard`
  leaves the section.

**Decision:** a thin `AnalysisSection` container rendering a `<nav>` sub-nav of buttons + the single
active view (lazily mounted). Six view components untouched; only `App.tsx`/`Dashboard.tsx` + the two
e2e specs change. No schema/API/domain change.

## What R3 built

### `AnalysisSection.tsx` (new, front-end only)

- Holds `tab` state (one of `spend`/`budget`/`forecast`/`credit`/`payoff`/`networth`), default `spend`
  (the prior "Analysis" button's destination).
- Renders `<nav aria-label="Analysis views">` of six `<button>`s; the active one carries
  `aria-current="page"`. Clicking a tab calls `setTab`.
- Below the nav, conditionally renders **only the active** view (the `? … : null` idiom used elsewhere),
  passing `api` + `onBack`. Switching unmounts the previous view and mounts the next.

### `App.tsx`

- Six `View` variants (`analysis`/`budget`/`forecast`/`credit`/`payoff`/`networth`) and their six route
  conditionals collapsed to a single `{ name: "analysis" }` → `<AnalysisSection api onBack />`.
- Six imports (`AnalysisView` … `NetWorthView`) replaced by one (`AnalysisSection`).
- Dashboard props: six `onOpen*` handlers collapsed to one `onOpenAnalysis`.

### `Dashboard.tsx`

- Five header buttons (Budget, Forecast, Credit, Payoff, Net worth) removed; the single **"Analysis"**
  button (→ `onOpenAnalysis`) remains. Five now-unused props dropped from the signature.

No other source files changed. **The six analysis view components are byte-for-byte untouched.**

- **SCA note:** `npm audit` reports 3 **high**-severity advisories (kysely chain); the gate is
  `--audit-level=critical`, so it passes (exit 0). Pre-existing, not introduced by R3.
- **Navigation changed:** the Dashboard no longer has **Budget / Forecast / Credit / Payoff / Net worth**
  buttons — they're tabs inside the **Analysis** section now. e2e/component selectors that clicked those
  header buttons must go via the section: `e2e/setup.ts` exposes `openAnalysis(page, tab)` (tab labels:
  `Spend`, `Budget`, `Forecast`, `Credit`, `Payoff`, `Net worth`). The Dashboard "Analysis" button still
  opens the section on the Spend tab.
- **Only the active analysis view is mounted** — switching tabs unmounts the previous one (re-fetch on
  re-show). Don't assume two analysis views are in the DOM at once.
- **testing-library `getByRole` has no `exact`** — a string `name` is already a full, exact accessible-name
  match (use a `RegExp` for substring). (Playwright's `getByRole` _does_ take `exact: true`.)
- **One `<main>` per page:** `AnalysisSection` is a fragment (`<nav>` + the view's `<main>`). Don't wrap it
  in another `<main>`.
- `reuseExistingServer: !process.env.CI` — kill stale `:3001`/`:5173` before e2e if you changed server
  code (R3 changed none).

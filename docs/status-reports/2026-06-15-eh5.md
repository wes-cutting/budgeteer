## What EH5 adds (and why it's the right minimal shape)

- **The missing layer, exactly.** The three existing layers — domain unit, API integration via
  Fastify `inject`, web component on jsdom against a **fake** API — never put a real browser in
  front of the real API. That's why a CORS misconfig shipped and was only caught by hand. EH5 closes
  that seam with the smallest real thing: one Chromium spec against both real servers.
- **The load assertion is the bug-catcher.** Step 1 asserts the `Budgeteer` heading renders **and**
  no `Couldn't load your data.` alert appears — i.e. the dashboard's opening `GET /accounts` +
  `GET /envelopes` actually completed cross-origin. A wrong CORS origin fails here the way the
  shipped bug did.
- **One real journey, end to end.** Account → envelope → deposit → allocation → **derived** envelope
  balance proves data → API → UI round-trips (real `POST`s + re-render), not just that a request
  returned 2xx.

## Design notes / decisions

- **Chromium only.** `npx playwright install chromium` (not Firefox/WebKit) — contains the
  footprint; the browser binary lives in `~/Library/Caches/ms-playwright`, never in the repo.
- **Separate `test:e2e`, not folded into `npm test`.** Keeps the Vitest inner loop fast and mirrors
  how `typecheck`/`lint`/`build` are each their own gate command. The documented gate is the
  sequence; e2e is one more step in it (README).
- **Dev server, not `preview`.** The web is served via `npm run dev` (`:5173`). Simplest, needs no
  prior build, and is the faithful reproduction of the CORS bug (a dev-origin issue). The production
  artifact is already gate-covered by `npm run build`. React StrictMode double-invokes the mount
  effect in dev → two load round-trips; harmless, Playwright auto-waits for the settled DOM.
- **Hermetic + reuse-safe.** No `DATABASE_URL` → in-process PGlite → each run starts from an empty,
  deterministic store. The spec also uses **stamped unique names** and asserts only on the rows it
  creates (never a global empty state), so it's robust even when `reuseExistingServer` reuses a
  dev server you already had up.
- **e2e TS is type-checked.** `tsconfig.e2e.json` is folded into `npm run typecheck`, so the new
  surface isn't an untyped escape hatch (the project's strong-typing rule).
- **One assertion needed an exact match.** `Balance: $500.00` (register header) vs. the
  ReconcilePanel's `Budgeteer balance: $500.00` both contain the substring; the header assertion
  uses `{ exact: true }`.

## Commands & gotchas (cold-start)

Gate is now: `npm run typecheck`, `npm run lint`, `npm test`, **`npm run test:e2e`**, `npm run
format`, `npm run build --workspace @budgeteer/web`.

- **One-time:** `npx playwright install chromium` (the browser binary isn't in the repo).
- `npm run test:e2e` **boots both servers itself** (API via `npm run start --workspace
  @budgeteer/api`, web via `npm run dev --workspace @budgeteer/web`) and tears them down. No manual
  setup.
- **Locally it reuses already-running servers** (`reuseExistingServer: !CI`). The spec's stamped
  names keep it passing against a populated dev DB; to mirror CI's hermetic run, stop local dev
  servers first.
- **The web must be served from `:5173`** — that origin is the API's `CORS_ORIGINS` default. Change
  the web port and the dashboard load fails (which the e2e will catch); update `CORS_ORIGINS` too if
  you do.
- CI (`.github/workflows/gate.yml`) stays **manual-only** (`workflow_dispatch`) — unchanged.

## Challenge findings (before building)

The kickoff flagged one risk: *does defaulting the register to the current month hide the
opening-balance transaction?* Resolved by reading `accountService.create`,
`transactionService.listByAccount`, and the `needs-allocation` route first:

- **The opening row is dated account-creation day** (`accountService` sets
  `occurred_on: todayStr()`). So for an account created in a **prior** month, a current-month
  default **would** hide it — and the opening row is the register's anchor (it's where the
  displayed balance comes from). **Decision: the register query always includes `kind='opening'`
  rows regardless of the window.** Cheap (≤1 per account) and keeps the bounded list
  self-explanatory. This special-cases the **register default**, exactly the option the kickoff
  floated.
- **"Needs allocation" surfacing is unaffected.** It is a separate household-wide endpoint
  (`GET /transactions/needs-allocation`) with **no** date filter, so an unallocated opening
  balance still surfaces there no matter the register window. No change needed there — this is
  what de-risks the whole concern.
- **Usability:** a date window with no UI to change it would trap the register on the current
  month forever. Added minimal **From/To date inputs** so the window is reachable. Search stays
  **client-side over payee/memo** per spec.

**Decision:** a read-path window + always-show-opening + client search. No domain change, no
new guard, no schema change.

## What R8 built

### `transactionService.listByAccount(accountId, opts?)` (service layer)

Added an optional `{ from?, to? }` window. When both are present, the query filters by
`occurred_on` **OR** `kind = 'opening'` (opening always survives the window); with no window,
every row is returned (backward compatible). Ordering unchanged (`occurred_on desc, created_at
desc`).

### Route — `GET /accounts/:accountId/transactions`

New `AccountTxnsRoute` generic (`Params` + `Querystring`). Reads `?from`/`?to`, defaults each to
the current month via the new `currentMonthRange()` helper in `util/dates.ts`, validates both
against `DATE_RE` → `400` on malformed input, then passes the window to the service.

### Frontend

**`apps/web/src/api.ts`:** `listTransactions(accountId, range?)` builds an optional `?from&to`
query string (interface + `httpApi`).

**`apps/web/src/test/fakeApi.ts`:** `listTransactions` applies the same window semantics
(opening always shown). Also aligned `makeTxn`/transfer defaults to **today** (the server already
defaults `occurredOn` to `todayStr()`), removing a latent hard-coded-date time-bomb that a
current-month default would have tripped.

**`apps/web/src/AccountRegister.tsx`:** `currentMonthRange()` default + `range`/`search` state;
`load()` passes the window; the effect re-fetches when `from`/`to` change; a labelled "Filter
transactions" form with **From date**, **To date**, and a **Search payee or memo** box; the list
renders the client-side-filtered `visible` rows with a "No transactions match your search." empty
state.

- **Date-fixture caution:** with the current-month default, tests that hard-code `occurred_on`
  must either use dates in the run-month or pass an explicit `?from&to` window. The existing
  "newest-first" test now passes `?from=2000-01-01&to=2100-12-31`; `fakeApi` defaults txn dates
  to **today** so jsdom tests stay in-window.

  runs, so a second `fill()` can be wiped. Wait for the first row to land before entering the
  second (see the search e2e).

  server code.

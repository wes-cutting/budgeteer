## Challenge findings (before building)

The kickoff leaned toward a **front-end fetch of the existing endpoint** over a **new lightweight count
aggregate**, and asked to confirm there's no N+1 / perf concern and that the badge stays current. I
checked the code and confirmed the lean:

- **Front-end fetch wins; the aggregate is unjustified ceremony.** `api.listNeedsAllocation()` already
  exists, is tested, and is the *same* endpoint the `NeedsAllocation` view calls on click. A count
  aggregate would add a service method + route + `06_API_CONTRACT` entry + tests to return a single
  integer — disproportionate for this scale (single household, manual entry).
- **No N+1 introduced.** `transactionService.needsAllocation()`
  ([`transactionService.ts`](../../apps/api/src/services/transactionService.ts)) is a **single
  round-trip** (one `selectTxns` + a batched `attachViews`) that then filters in JS. R2 adds **one
  parallel request** on Dashboard mount and reuses that exact query — it does not worsen the existing
  fetch-all-then-filter shape (a pre-existing, bounded characteristic).
- **Freshness is automatic — no cache wiring needed.** `App.tsx` conditionally renders the Dashboard
  keyed on `view.name` (no router keep-alive), so navigating to "Needs allocation", completing an
  allocation, and clicking **← Dashboard** *remounts* the Dashboard → its `useEffect` re-fetches → the
  badge is current. Documented as "refreshes on next Dashboard mount."
- **One improvement over the naive plan: graceful degradation.** The badge loads **independently** of the
  core `listAccounts/listEnvelopes` `Promise.all`. A needs-allocation failure leaves the badge silently
  absent instead of triggering the page-level `loadError` and blanking the Dashboard. The badge is
  auxiliary; the accounts/envelopes are the core.
- **Accessibility is name-based.** The count goes in the button's `aria-label`, not color alone. The
  visible pill uses safe contrast and the existing Dashboard-with-data axe scan exercises it.

**Decision:** a front-end fetch of `listNeedsAllocation()` in `Dashboard.tsx`, loaded independently, with
the count in the button's `aria-label` and a high-contrast pill. No new endpoint, no schema change, no new
domain primitive.

## What R2 built

### `Dashboard.tsx` (front-end only)

- **State + independent fetch:** new `needsCount: number | null` state; the mount `useEffect` calls
  `api.listNeedsAllocation()` *outside* the core `Promise.all` and sets `needsCount` to the list length.
  On error the badge stays absent (no `loadError`).
- **Badge render:** when `needsCount !== null && needsCount > 0`, the "Needs allocation" button gets
  `aria-label="Needs allocation (N)"` and renders a `<span style={BADGE} aria-hidden>` pill showing `N`.
  When the count is 0 (or not yet loaded), the button is the plain "Needs allocation" with no pill.
- **`BADGE` style:** a rounded pill, `#991b1b` background / white text (~8.3:1 contrast).

No other files changed (no API, no schema, no domain).

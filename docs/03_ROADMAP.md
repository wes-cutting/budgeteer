**Current focus:** **`R13` ‚úÖ done (275 Vitest + 47 e2e, gate-green) ‚Äî completes the DX backlog (`R10`‚Äì`R13`).**
Structured API logging: `buildServer`'s `logger` option is widened from `boolean` to the Fastify logger option
itself, and `index.ts` now passes `{ logger: { level: config.LOG_LEVEL } }` so the API emits Fastify's
bundled-pino request/response/error lines at the configured level. `LOG_LEVEL` (already in `config.ts`) is
tightened to a **closed enum** of pino levels ‚Äî validated at the boundary so a typo fails loudly at startup
(spine ¬ß8) instead of throwing deep inside pino. Tests omit the option ‚Üí `logger: false` ‚Üí quiet + deterministic
(unchanged). **Challenge resolved:** the risk was leaking financial request bodies/headers into logs ‚Äî verified
Fastify's **default** serializers log only `method`/`url`/`statusCode`/`responseTime` (+ error type/message/stack),
**never** bodies or headers, so we keep them as-is and add **no** body serializer (SECURITY.md ¬ß1/¬ß5). The
regression net is a capture-stream test asserting **both** halves: the access line is present *and* a money-route
request's `memo`/`payee`/amount + an `authorization` header never appear in the captured output. A real-startup
smoke confirmed structured JSON to stdout with a `reqId` correlation id and no body leak. **Zero API-contract /
schema / domain change** ‚Äî logging is a cross-cutting effect, not a route. **Next: roadmap DX + UX backlog is
clear; remaining work is the Hardening track and new feature slices.**

## 1. How to use this roadmap

The plan of record, kept live. **Top = next.** Budgeteer is a **focused product/app** (the
full-kit row in [¬ß11](00_WAYS_OF_WORKING.md)), so it runs the full chain ‚Äî but every *build*
item below is a **vertical, usable slice** that passes the gate before it's `Done`; there
are no "layer" items. Item statuses track *delivery*; the `Proposed/Validated/Accepted`
ladder tracks *documents* ‚Äî don't conflate them.

**Item status vocabulary:** `Planned` ¬∑ `Ready` ¬∑ `In progress` ¬∑ `Done` ¬∑ `Deferred` ¬∑ `Dropped`.

## 2. Sequencing model

Adapted from [¬ß7](00_WAYS_OF_WORKING.md), justified by what discovery + SPIKE-01 already
retired:

1. **Decisions & feasibility first.** The value/UX bet is de-risked (SPIKE-01); the
   *unretired* risk is the **unchosen stack** + unproven technical feasibility. So a tight
   **feasibility spike (SPIKE-02)** runs before the foundation, producing the stack ADRs.
2. **Foundation slice** ‚Äî a usable shell (accounts + envelopes you can create) to build the
   core loop into. Security guardrails are set here, day-zero (see note).
3. **Domain slices, validated heart first** ‚Äî the core enter‚Üíallocate loop (closes the
   SPIKE-01 felt-friction caveat), then the accelerators that make the paycheck painless,
   then the further capabilities.
4. **Analysis**, once the core loop produces real data to analyze.
5. **Hardening**, once there's real data/usage to measure against.
